# 21 工程纪律 — 工具不是纪律的替代品

> **一句话**：能写成工具的纪律只能解决"不知道怎么做"的问题，解决不了"知道该做什么但没做"的问题。

## 背景

这个项目有详尽规则：
- `CLAUDE.md` 写了三层测试策略（快/慢/全量标记）
- `docs/01-guides/ci/CHECKS.md` 写了"日常改 module 只跑相关测试"
- 连 pre-push hook 都配置了自动快子集

但 AI 仍然反复犯一个错：**提交前无脑跑全量测试**。即使只改了 `src/utils/` 4 个文件，也要跑 943 个测试，白白浪费 12 分钟。

## 根本原因

### 1. 规则在文档里，判断在脑子里

```
写进文档的规则 ← 是"知识"
执行时的决策 ← 是"行为"
```

知识可以靠写文档传递，行为需要**每次执行前主动判断**。AI 和人类的区别是：AI 不会自动把知识转化成行为判断——它需要被提示。而项目的规则文档在 Context 里，AI 有它但不一定想到要去查。

### 2. 默认行为 = "最安全的那个"

当我不知道"该跑多少测试"时，我的默认行为是"全量最保险"。这个默认值是错的：

| 场景 | 正确做法 | 我的默认 |
|:-----|:---------|:---------|
| 改 utils | 跑 `tests/test_utils/`（47 tests, 6s） | 跑全部（943 tests, 12min） |
| 改 debate | 跑 `tests/test_debate/` | 跑全部 |
| 简单任务 | 用 deepseek-chat 快速响应 | 想切 deepseek-v4-pro |

"不知道就选最重的那条"是一种防御性思维，但在这个项目里，**正确的默认是"按范围选最轻的"**。

### 3. 工具解决不了"忘了用工具"的问题

最终解决方案是写了 `scripts/check.py`——自动检测变更范围，智能选测试。但这揭示了一个更深的问题：

> 如果一个人（或 AI）总是忘记查文档再行动，给他一个工具他也会忘记用。

写工具只是第一步。真正要改的是**行动前的停顿习惯**。

## 项目中的真实代码

本会话创建的两个关键记录：

### 教训的直接产物：`scripts/check.py`

```python
def pick_test_targets(changed_files: set[str]) -> list[str] | None:
    """根据变更文件决定跑哪些测试。"""
    # 只改了 src/utils/ → 只跑 tests/test_utils/
    # 改了跨模块文件 → 跑全量子集
    # 只改了 docs/ → 跳过测试
```

这是把"判断"固化成工具的结果。但注意：工具只是把已知规则写进代码，它本身不创造新知识。

### 教训的间接产物：本会话的两次提交

```
c1dc629  feat: DP-001 + scripts/check.py  ← 写了工具但没更新文档
36d1b85  docs: check.py 流程闭环          ← 用户指出后才补的
```

第 1 次提交只有工具，没有更新任何引用文档。第 2 次提交才把 24 处文档统一指向新工具。

**写了新东西 → 但不更新所有引用它的地方**——这不是新问题，这是这个项目反复出现的"五同步"（代码+测试+文档+债务+引用清理）中引用清理环节缺失的同一个模式。

## 自己试试（5 分钟）

1. 读一下 `scripts/check.py` 的 `pick_test_targets()` 函数
2. 想一想：如果你改了 `src/agents/` 的一个文件，check.py 会怎么选测试？
3. 然后手动 `python scripts/check.py` 看看实际输出
4. 最后运行 `python scripts/check.py --full` 对比时差

## 三个层面的教训

### 第一层：行为层 — 每次行动前先判断

- 不是所有改动都需要全量测试
- 不是所有任务都需要最强模型
- 不是所有问题都需要全代码库搜索

停下 3 秒问自己：**"这个改动范围多大？对应做什么就够？"**

### 第二层：机制层 — 工具要闭环

写了一个新工具/新规则后，反问自己：
- 所有引用旧做法的地方更新了吗？
- 旧文件删除了吗？
- 文档中最少几处是错的？

check.ps1 删除前的状态：check.py 写好了，但 24 处文档还写着 `make check`。

### 第三层：元认知层 — 知道"自己不知道"

最可怕的是"不知道自己没在做正确的事"。这次用户指出来之前，我没有意识到自己在无脑跑全量测试。需要把"我现在的决策依据是什么"变成一个外部检查点。

**建议**：每次执行 `check.py` 前，看它的输出——它选的测试目录合不合理？这个简单的外部反馈循环就是元认知的锚点。
