###### TD-061 结果参数回调引擎缺失 — 结果来了，大部分相关策略不响应

| 属性 | 值 |
|------|-----|
| **分类** | `架构设计` `severity:critical` `module:all` `impact:功能缺失` |
| **发现日期** | 2026-06-23 |
| **发现人** | 用户审视 |
| **状态** | `🔧 修复中`（RC-001/RC-002 已完成，用户行为和风控回调待接入） |
| **本金估算** | ∼3-4天（核心引擎+三大回调实现） |
| **日利息** | 各大师对特定板块的误判永远不会被校准；用户的个人行为经验永远进不了辩论；风控策略不会根据近期结果自适应 |
| **实盘影响** | **不修则无法形成"投研闭环"** — AI 建议再好，用户操作不听也不知道为什么。系统永远学不到用户的习惯和偏差 |
| **触发场景** | 每次用户决策后、每次市场结果出来后 |
| **用户能发现吗** | 能 — 体现在"AI 永远在重复同样的错误"、"理财经理从用户的操作中学习不到东西" |

**描述**：
M2 反思（AI vs 市场）和 M3 信任度（大师权重统计）都只覆盖了"结果反馈"的一个极薄子集。缺少统一的"结果参数回调引擎"——当任一结果到达（市场结果、用户操作结果），自动触发全链路策略参数校准。

**具体问题**：
1. ❌ M3 只调大师投票权重——不调大师对特定板块的置信度基准、不调分析师层指标权重、不调风控参数
2. ❌ 用户的操作（买/卖/持有/放弃）完全没被追踪——不知道用户是否采纳了 AI 建议、用户的实际盈亏如何、用户的决策模式是什么
3. ❌ 没有统一的 `ResultCallbackEngine`——每个需要响应结果的模块都得自己"等结果"，没有注册-订阅-分发机制

**利息分析**：
- 日复一日，系统不会越来越懂用户的个人习惯——永远处于"冷启动"状态
- 某大师在科技股连续亏 → 权重只会轻微下调，但它在科技股发言时依然 100% 置信度——应该先校准它在科技股场景下的可信度
- 用户每次在 AI 说"持有"时提前卖出 → 系统永远学不到用户的这个倾向

**修复方向**：
设计一个 `ResultCallbackEngine` 统一接收结果事件 → 分发到注册的回调处理器 → 各回调按需调整策略参数。同时扩展 M3 的 `AgentOutcome` 到按板块/场景细分统计。

**进展记录**：
- 2026-07-10：RC-001 核心分发器完成。`src/callback/engine.py` 已支持事件分发、回调注册、优先级、冷却、错误记录、自动禁用、执行记录持久化；`tests/test_callback/test_engine.py` 覆盖 5 条核心路径。
- 2026-07-10：RC-002 回调层完成。`AgentOutcome.sector`、按板块胜率、`m3_ext` 结果回调已上线。
- 2026-07-13：RC-002 编排器 dispatch 完成。`DebateOrchestrator.reflect_on_decision()` 收到 `ActualOutcome` 后分发 `ACTUAL_OUTCOME_RECEIVED`，M3-EXT 会把历史 `agent_analyses` 与实际结果写入 TrustTracker；复盘看板入口仍待接入。
- 尚未关闭：RC-003 用户行为追踪、RC-004 风控参数自适应、R4 交易复盘等业务闭环尚未接入，系统还不能完整按用户操作和实际盈亏自动学习。
