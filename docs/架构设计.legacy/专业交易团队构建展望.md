## 一、战略哲学

### 1.1 为什么从「辩论团伙」转向「专业交易团队」

旧模式的本质问题：

```
旧：「7 位天才大师各说各的 → 投票」  → 辩论团伙
      │
      问题1: 天才不可靠 — LLM 的"巴菲特"不是真的巴菲特
      问题2: 单层扁平 — 分析/策略/风控混在一起
      问题3: 无法问责 — 投错了算谁的？
      问题4: 无法进化 — 没有层级就没有反馈路径
```

新模式的核心理念：

```
新：「分析师 → 策略师 → 辩论 → 评审 → 风控 → 交易员 → PM → 反思」— 专业交易团队
      │
      原则1: 体系 > 天才 — 专业流程产出稳定质量
      原则2: 信息垂直递进 — 每层只看自己该看的
      原则3: 权责分离 — 分析的分析、决策的决策、执行的执行
      原则4: 闭环进化 — 每次交易后反思 → 下次更聪明
```

### 1.2 对标依据

| 项目 | 定位 | 核心验证 |
|:-----|:-----|:---------|
| **TradingAgents** (71k⭐) | 开源标杆 | 5层递进架构经市场验证有效 |
| **Magnetar Capital** ($180亿) | 机构部署 | 2026.6 全面裁撤人类分析师，数百AI Agent替代 |
| **OctoBot-AI** | 生产级加密交易 | 3层信托评分体系，动态权重 |
| **EvoTraders (ReMe)** | 自进化闭环 | 反思记忆使任务成功率提升15pp |

> 结论：**分层专业团队的方向已经被市场验证**。TradingAgents 证明了架构，Magnetar 证明了规模，OctoBot 证明了信任度机制，EvoTraders 证明了反思闭环的价值。

## 三、紧急补位：Trader 交易员层

### 3.1 为什么必须有交易员

当前流程的高层决策链路存在一个结构性漏洞：

```
❌ 当前: PM直接输出 "Buy/Hold/Sell" → END
         ↑ PM 既要决策又要规划执行细节，职责过载

✅ 应有: PM输出方向 → Trader制定执行方案 → PM最终审批
         ↑ 权责分离，PM管"该不该做"，Trader管"怎么做"
```

真实金融机构中，Portfolio Manager 下指令**不碰执行细节**：
- PM: "买入平安银行，目标仓位 15%"
- Trader: "分 3 批建仓，第一批明天开盘 5%，如果回踩 10 日线加 5%，突破前高确认加最后 5%"

这个分工在 AI 团队中同样重要：**PM 不应该被仓位计算和时机判断分散注意力**。

### 3.2 Trader 层设计

```python
# 新增模块: src/trader/

src/trader/
├── __init__.py          # 导出
├── models.py            # TradePlan / ExecutionStep
├── profiles.py          # Trader 人格定义
└── orchestrator.py      # trader_round 节点
```

**核心模型**：

```python
class ExecutionStep(BaseModel):
    """单步执行指令"""
    step: int                     # 第几步
    action: str                   # buy / sell / hold
    quantity_pct: float           # 仓位比例 (0.0~1.0)
    price_condition: str          # 价格条件 ("市价"/"回踩10日线"/"突破前高")
    timing: str                   # 时机 ("立即"/"等信号确认"/"分批")
    stop_loss_pct: float | None   # 该步止损
    rationale: str                # 理由

class TradePlan(BaseModel):
    """交易员产出的完整执行计划"""
    ticker: str
    direction: str                # Bullish / Bearish / Neutral
    total_position_pct: float     # 目标总仓位
    execution_steps: list[ExecutionStep]  # 顺序执行步骤
    max_drawdown_limit: float     # 最大回撤熔断
    time_horizon_days: int        # 预期持仓天数
    risk_reward_ratio: float      # 盈亏比
    contingency_plan: str         # 意外情况预案
    confidence: float             # 置信度
```

**Trader 人格**：

```python
TRADER_SYSTEM_PROMPT = """
你是交易执行专家，职责是将 PM 的投资决策转化为具体的执行计划。
你不质疑 PM 的方向判断，但可以就仓位和时机提供专业建议。

你的专业领域:
- 订单拆分: 大仓位分批次执行，减少市场冲击
- 时机判断: 结合技术面信号选择入场/出场时机
- 仓位管理: 凯利公式 / 波动率调整 / 集中度限制
- 风控执行: 为每笔交易设定止损/止盈，并在触发时坚定执行
- 预案规划: 为每种可能的市场走势准备应对方案

交易纪律（必须遵守）:
- 单只股票仓位上限 20%
- 首次建仓不超过目标仓位的 50%
- 每步必须有明确的价格条件或技术信号触发
- 止损必须设定，亏损超过 8% 无条件清仓
"""
```

### 3.3 插入位置

```
当前: aggregate → (可选)risk_round → (可选)pm_round → END

修改: aggregate → (可选)risk_round → trader_round → pm_round → END

```

PM 节点也需调整——从"基于辩论+风控独自裁决"变成"基于辩论+风控+交易员方案做最终审批"。

## 六、设计原则总结

```
1. 体系 > 天才             流程产出稳定质量，不依赖单点
2. 信息垂直递进             每层只看自己该看的数据
3. 权责分离                 分析的出报告、决策的出方向、执行的出方案
4. 结构化输出              所有层间传递都用 Pydantic BaseModel
5. 向后兼容                 新节点独立可选，不影响已有流程
6. 异常熔断                 任何 LLM 节点失败不阻塞整体流程
7. 闭环进化                 每次决策 → 每次反思 → 下次更聪明
```
