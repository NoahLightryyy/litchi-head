### 3. 工作流体系判断

**总体判断**：继续沿用当前工作流，但按任务类型压缩执行粒度。

当前工作流的优点：
- “五同步”是必要纪律，尤其适合实盘辅助项目。
- 部门化文档能避免跨模块工作时丢上下文。
- 强制闸门能防止“代码写完但日志/测试/债务没同步”的坏习惯。
- `scripts/check.py` 的智能检查比旧式全量测试更适合日常节奏。

当前工作流的风险：
- 对小任务和快速审视来说，完整收尾偏重，容易产生仪式性开销。
- 文档入口很多，如果每次都全量同步，容易把开发节奏拖慢。
- “每次代码改动更新全局 HANDOVER/README/ROADMAP”的触发条件需要谨慎执行：状态没变时应只记工作日志，避免文档 churn。

建议执行策略：
- **架构/跨模块/实盘风险任务**：继续走完整模式，严格五同步。
- **模块内小修/快速审视/状态判断**：走轻量模式，只更新工作日志，必要时同步相关部门文档。
- **涉及数据真实性、置信度、风控、交易记录**：无论改动大小，都按完整模式处理。
- **功能交付后**：继续执行强制闸门，但测试只跑相关模块；全量测试只在跨模块基础设施或用户明确要求时执行。

### 6. Codex Batch Loop 自动化协议

**用户需求**：关闭聊天窗口后，希望后续能用自动化 loop 继续推进，而不是每次重新解释上下文；同时希望明确 `computer-use` Skill 的用途。

**同步结论**：
- 新增 `docs/01-guides/LOOP.md`，定义“启动 litchi-head loop / 继续 loop / 收尾 loop”三类入口。
- Loop 单轮固定执行：恢复上下文 → 判断优先级 → 拆原子任务 → 实现 → 验证 → 落文档 → 汇报。
- Codex App 的定时/心跳自动化用于周期触发 loop，需要用户指定节奏后再创建。
- `computer-use` 只用于 Windows GUI 验证，例如打开浏览器检查前端真实显示；不用于读写文件、跑测试、Git 或替代代码主循环。

### 7. Loop 收尾与上下文续接边界

**用户需求**：希望 `computer-use` 在上下文用完时打开新的窗口继续工作，并要求每次做完一个功能就收尾保存。

**同步结论**：
- `computer-use` 不能用于自动化 Codex 桌面 App 或 Codex CLI，因此不能让它去点击/打开新的 Codex 聊天窗口。
