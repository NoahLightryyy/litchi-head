## 完成内容

### CI 治理部独立（~20min ✅）

**问题**：CI 归属基础设施部，但 CI 治理（门禁标准、根因库、问题追踪）与基础设施工具链（LLM/配置/日志）是两套职责。HANDOVER.md 中 CI 状态需要一大段描述，跨部门引用混乱。

**方案**：成立第 11 个部门（CI 治理部），明确边界。

#### 管什么 ✅

| 范围 | 说明 |
|:-----|:------|
| CI 工作流配置 | `.github/workflows/ci.yml` 的步骤、顺序、超时、矩阵策略 |
| CI 门禁标准 | 什么算绿/红，绿/黄/红分别怎么处理 |
| CI 根因知识库 | `TROUBLESHOOTING.md` 各类型 CI 失败排查与修复 |
| CI 问题追踪 | `ISSUES.md → DEBT.md` CI 系统本身的问题 |
| 本地拦截规范 | pre-push hook 规则、`make check` 执行标准 |
| CI 流程优化 | 跑慢了、误报了、该加新检查了 |

#### 不该管 ❌

| 边界 | 归属部门 |
|:-----|:---------|
| GitHub Actions runner 可用性/费用 | 基础设施部 |
| 代码风格规则选哪些（ruff 配置） | 基础设施部 |
| 测试框架配置（pytest asyncio_mode） | 基础设施部 |
| 具体代码质量修复 | 各业务部门自己修 |

#### 核心原则

> CI 治理部是"质量门禁"的制定者和维护者，不是"质量修复"的执行者。
> CI 红了 → 定位是哪一层的锅 → 通知对应部门修 → 跟踪到闭环。
