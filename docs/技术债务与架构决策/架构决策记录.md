## 1. ADR 规范

### 1.1 什么时候写

以下情况**必须**记录 ADR：
- 引入新的外部依赖（库/框架/服务）
- 模块间的接口协议约定
- 数据存储方案选型
- 重大架构重构
- 舍弃了一个明显的替代方案

### 1.2 怎么写

每条 ADR 包含：

```markdown
## ADR-NNN: 标题

| 字段 | 值 |
|------|-----|
| 日期 | YYYY-MM-DD |
| 状态 | ✅ 已采纳 / 🔄 提议中 / ❌ 已废弃 / ⏪ 已替换 |
| 影响范围 | 受影响的模块/组件 |

**上下文**：
做这个决策的背景和驱动力。

**决策**：
最终选定的方案。

**理由**：
1. 选它的原因
2. ...

**舍弃方案**：
- 方案A：淘汰理由
- 方案B：淘汰理由

**后果**：
- 正面：...
- 负面：...（这就是引入的技术债务）
```

### 1.3 状态说明

| 状态 | 含义 |
|------|------|
| ✅ 已采纳 | 当前有效的决策 |
| 🔄 提议中 | 正在讨论，尚未决定 |
| ❌ 已废弃 | 明确放弃，不再考虑 |
| ⏪ 已替换 | 曾经有效，但已被新 ADR 取代 |

## 3. 决策记录

### ADR-001: Pydantic 作为全栈数据校验层

| 字段 | 值 |
|------|-----|
| **日期** | 2026-06-03 |
| **状态** | ✅ 已采纳 |
| **影响范围** | 全项目 |

**上下文**：
多 Agent 系统需要跨模块传递大量结构化数据，需要统一的序列化/校验方案。
同时，LLM 的结构化输出（`with_structured_output`）原生支持 Pydantic 模型。

**决策**：
所有模块间的数据传输统一使用 Pydantic `BaseModel`：
- 通信协议 `AgentMessage` — Pydantic
- Agent 输入/输出 — Pydantic
- 配置管理 — Pydantic Settings

**理由**：
1. 与 LLM `with_structured_output()` 天然对接
2. 零成本序列化/反序列化（`.model_dump()` / `.model_validate()`）
3. JSON Schema 导出能力方便调试和文档生成
4. Python 3.12 标准依赖，无额外引入

**舍弃方案**：
| 方案 | 舍弃理由 |
|------|---------|
| `dataclasses` 纯数据类 | 缺少校验和序列化能力 |
| `msgspec` | 性能更好但生态不如 Pydantic 与 LangChain 集成深 |
| Apache Avro / Protobuf | 太重，不适合 MVP 阶段 |

**后果**：
- ✅ 统一的数据契约层，减少模块间的沟通成本
- ✅ 调试友好（JSON Schema 可直接可视化）
- ⚠️ 性能不如 msgspec（但 MVP 阶段无感知）

### ADR-002: LangGraph 作为 Agent 编排框架

| 字段 | 值 |
|------|-----|
| **日期** | 2026-06-03 |
| **状态** | ✅ 已采纳 |
| **影响范围** | `src/core/`, `src/debate/` |

**上下文**：
需要框架来编排多个 Agent 的工作流，支持串行/并行/条件分支和状态共享。

**决策**：
使用 LangGraph 的 `StateGraph` 作为编排引擎。

**理由**：
1. 图结构天然适配 Agent 工作流（节点 = Agent，边 = 消息传递）
2. LangChain 生态原生支持
3. 内置并行节点和条件路由
4. 社区活跃，文档充足

**舍弃方案**：
| 方案 | 舍弃理由 |
|------|---------|
| AutoGen（微软） | 多 Agent 对话能力强但状态管理不够灵活 |
| CrewAI | 更上层封装但可控性不如 LangGraph |
| 自定义状态机 | 灵活但重复造轮子 |
| Temporal / Airflow | 太重，适合后端服务不适合 Agent 编排 |

**后果**：
- ✅ 得益于 LangGraph 生态，后续 LLM 集成省大量工作
- ⚠️ 版本更新快，需锁定版本

### ADR-008: Agent 数据契约与通信协议规范

| 字段 | 值 |
|------|-----|
| **日期** | 2026-06-07 |
| **状态** | ✅ 已采纳 |
| **影响范围** | `src/agents/`, `src/core/`, `src/utils/`, 数据库设计, 所有 Agent 实现 |

**上下文**：

现有 `AgentResult.data: dict` 和 `AgentMessage.payload: dict` 是"任意 JSON"类型，意味着：

1. ❌ **无法静态校验** — 下游消费方不知道 data 里有什么字段，Pyright 零保护
2. ❌ **无法建表** — 未来写数据库时，每个 Agent 的 data 结构不同，无法展开为列
3. ❌ **接口腐化** — 新增 Agent 时没有契约约束，结构任意演化
4. ❌ **序列化陷阱** — `datetime` 字段默认不被 `json.dumps` 支持

同时，LangGraph 编排器的 State 尚未定义，如果直接用 `TypedDict` 起步，未来改 Pydantic 时会迁就大量代码。

**决策**：

全栈采用 **Pydantic 契约驱动** —— 每一笔数据在诞生时就带着明确的 Schema。

#### 1. 每个 Agent 定义自己的 I/O 模型（强制）

```python
# src/agents/output_models.py 或每个 Agent 文件内
class NewsOutput(BaseModel):
    """新闻分析 Agent 的输出合约 —— 同时也是 DB 表字段定义"""
    summary: str = Field(..., max_length=500, description="新闻摘要")
    sentiment: Literal["positive", "negative", "neutral"]
    sentiment_score: float = Field(..., ge=0.0, le=1.0)
    entities: list[dict] = Field(default_factory=list)
    key_points: list[str] = Field(default_factory=list)

class MarketOutput(BaseModel):
    """行情分析 Agent 的输出合约"""
    ticker: str
    price_current: float
    change_pct: float
    technical_indicators: dict = Field(default_factory=dict)
    analysis: str = ""
```

**规则**：
- 每个 Agent 必须至少定义一个输出 Pydantic 模型（输出合约）
- 可选定义输入 Pydantic 模型（输入合约）
- 输出模型必须带有 `Field(description=...)` → 自动生成文档/DB schema
- 禁止向 `AgentResult.data` 写入无结构的任意 dict

#### 2. AgentResult 泛型化（关闭 TD-002）

```python
T = TypeVar("T", bound=BaseModel | None)

@dataclass
class AgentResult(Generic[T]):
    agent_name: str = ""
    session_id: str = ""
    success: bool = True
    data: T = None  # type: ignore  # 类型化输出
    confidence: float = 0.0
    reasoning: str = ""
    error: str | None = None
    latency_ms: float = 0.0

    # ... 见 §2a 实际实现
```

> **✅ 2026-06-07 已实现**（实际使用 Pydantic BaseModel，见 §2a）

**收益**：
- `AgentResult[NewsOutput]` — 下游明确知道 data 是 NewsOutput 类型
- Pyright 可校验：`result.data.summary` ✅ / `result.data.ticker` ❌（编译报错）

#### 2a. AgentResult 实际实现（2026-06-07）

```python
# src/agents/base.py 实际落地代码
T = TypeVar("T")

class AgentResult(BaseModel, Generic[T]):
    """Pydantic + 泛型，与 ADR-001 统一"""
    agent_name: str = ""
    session_id: str = ""
    success: bool = True
    data: dict | T = Field(default_factory=dict)  # dict 向下兼容
    confidence: float = 0.0
    reasoning: str = ""
    error: str | None = None
    latency_ms: float = 0.0
```

**与设计差异**：

| 维度 | 设计 | 实际实现 | 理由 |
|------|------|---------|------|
| 基类 | `@dataclass` + `Generic[T]` | `BaseModel` + `Generic[T]` | ADR-001 统一，自带 `model_dump()` |
| data 默认值 | `T = None` | `dict \| T = Field(default_factory=dict)` | 向后兼容全部 88 个测试 |
| 验证 | — | 88 tests ✅ + 新增 2 个 Generic 测试 | 零回归 |

#### 3. 编排器 State 使用 Pydantic BaseModel

```python
class DebateState(BaseModel):
    """LangGraph 共享状态 —— 数据总线
    
    映射关系：
      DebateState         → DB sessions 表
      agent_outputs[name] → DB agent_outputs 表
      messages[]          → DB messages 表
    """
    session_id: str
    input_data: dict = Field(default_factory=dict)
    current_step: str = "start"
    agent_outputs: dict[str, Any] = Field(default_factory=dict)
    messages: list[AgentMessage] = Field(default_factory=list)
    started_at: str = Field(default_factory=lambda: datetime.now().isoformat())
    completed_at: str | None = None
    error: str | None = None

    model_config = ConfigDict(
        json_encoders={datetime: lambda v: v.isoformat()},
    )
```

#### 4. AgentMessage 序列化就绪

```python
class AgentMessage(BaseModel):
    # ... 现有字段 ...
    payload_schema: str = ""  # ← 新增：记录 payload 结构的 schema ID

    model_config = ConfigDict(
        json_encoders={datetime: lambda v: v.isoformat()},
    )

    def to_db_record(self) -> dict:
        """直接可写入数据库的一行"""
        return self.model_dump(mode="json")
```

#### 5. SessionRecord 辩论快照

```python
class SessionRecord(BaseModel):
    """完整辩论会话记录 —— 结束时可 dump 存盘"""
    session_id: str
    status: Literal["running", "completed", "failed"]
    created_at: str
    completed_at: str | None = None
    outputs: dict[str, dict] = Field(default_factory=dict)
    messages: list[dict] = Field(default_factory=list)
    conclusion: dict | None = None

    def to_db_json(self) -> str:
        return self.model_dump_json()
```

**理由**：

1. **与 ADR-001（Pydantic 全栈）一脉相承** — 不是引入新框架，是把现有约定硬化
2. **DB 迁移零成本** — 每个 Pydantic 模型的字段定义=CREATE TABLE 的 DDL
3. **LangGraph 原生支持** — `StateGraph(state_schema=DebateState)`，State 自动校验
4. **API 响应结构一致** — 见 [common/patterns.md](../common/patterns.md) API 响应格式模式

**舍弃方案**：

| 方案 | 舍弃理由 |
|------|---------|
| 继续用 `data: dict`，+1 条债务 | 利息已到上限，Phase 0 不处理 Phase 1 就来不及 |
| Protobuf / Avro 作为契约 | 太重，Pydantic 足够且与 LangChain 生态集成 |
| 数据库先 "JSON 乱炖" 后面再拆 | 后面拆数据是出了名的高成本操作，不如一步到位 |

**后果**：
- ✅ 新 Agent 开发者一眼知道要输出什么字段
- ✅ 未来 DB 迁移：Pydantic 模型 → SQLAlchemy 模型映射自然
- ⚠️ 每个 Agent 多写 ~15 行模型定义（可接受，写一次受益终身）

**关联技术债务**：
- TD-002 — `AgentResult.data` 泛型：本 ADR 直接修复
- 本 ADR 无新增债务

### ADR-009: MCP 工具扩展架构

| 字段 | 值 |
|------|-----|
| **日期** | 2026-06-07 |
| **状态** | ✅ 已采纳 |
| **影响范围** | `src/agents/base.py`, `src/core/orchestrator.py`, 所有 Agent 实现 |

**上下文**：

未来各 Agent 需要接入外部工具来获取数据：

| Agent | 未来需要的工具 | 可能的 MCP 服务 |
|-------|--------------|----------------|
| 🗞️ 新闻分析 | 新闻搜索、RSS 订阅、舆情监控 | NewsAPI / akshare / 自定义爬虫 |
| 📈 行情分析 | 实时行情、K 线、技术指标 | akshare / 同花顺 / Yahoo Finance |
| 🎓 教育小智 | 金融知识库、计算器 | RAG 向量库 / Wikipedia |
| 🧙 大师 Agent | 大师语料库、财务数据查询 | RAG 向量库 / 财务数据 API |

Phase 0 时不接入真实的 MCP 服务（避免外部依赖），但架构必须预留扩展点。

**决策**：

#### 1. BaseAgent 新增 `get_tools()` 扩展点

```python
class BaseAgent(ABC):
    """所有 Agent 的抽象基类"""

    def __init__(self, name: str, config: dict | None = None):
        self.name = name
        self.config = config or {}
        self.logger = AgentLogger(f"agent.{name}")

    @abstractmethod
    async def run(self, ctx: AgentContext) -> AgentResult:
        """Agent 核心逻辑"""
        raise NotImplementedError

    def get_tools(self) -> list[Any]:
        """声明 Agent 所需的工具集
        
        Phase 0 默认返回空列表（无外部工具依赖）。
        Phase 1+ 子类重写此方法返回具体的 Tool/MCP 工具。
        
        编排器会在运行 Agent 前将工具注入到 LLM 调用上下文。
        """
        return []

    async def run_safe(self, ctx: AgentContext) -> AgentResult:
        """run 的安全包装（超时+异常）"""
        # ... 现有逻辑不变 ...
```

**Phase 0 默认行为**：
- `get_tools()` 返回 `[]` — Agent 纯 LLM 推理，无外部工具
- 编排器不做工具注入 — 保持简单

**Phase 1+ 扩展行为**（示例，暂不实现）：

#### 2. MCP 接入架构（未来）

```

  │

  │     │
```

#### 3. 编排器中的工具注入逻辑（Phase 1+ 实现计划）

```python
class DebateOrchestrator:
    async def _run_agent_node(self, state: DebateState, agent: BaseAgent) -> dict:
        """在 StateGraph 节点中运行 Agent，自动注入工具"""
        ctx = AgentContext(
            session_id=state.session_id,
            input_data=state.input_data,
            memory={},
            config={},
        )
        
        # Phase 1+：如果有工具，绑定到 LLM
        tools = agent.get_tools()
        if tools:
            # 工具注入逻辑（预留）
            pass
        
        result = await agent.run_safe(ctx)
        return {"agent_outputs": {agent.name: result.data}}
```

**理由**：

1. **最小侵入** — `get_tools()` 是空列表返回，不改变 Phase 0 任何行为
2. **渐进增强** — 子类按需重写，不影响其他 Agent
3. **与 LangGraph ToolNode 对齐** — langgraph-prebuilt 1.1.0 已安装，方案一致
4. **与 ADR-002 (LangGraph) 一致** — 编排器是自然注入点

**舍弃方案**：

| 方案 | 舍弃理由 |
|------|---------|
| Phase 0 就接 MCP 服务 | 外部依赖过早，徒增复杂度 |
| 不用 `get_tools()`，直接在 `run()` 里写死 | 耦合度高，后续迁移到 MCP 要改 Agent 核心逻辑 |
| 统一的全局工具注册中心 | 过度设计，YAGNI — 等有 3+ 个 Agent 需要工具再说 |

**后果**：
- ✅ Phase 0 Agent 代码纯净无外部依赖
- ✅ 未来接入 MCP 时只需重写 `get_tools()` + 增加对应 MCP 客户端
- ⚠️ 增加了 1 个空方法 `get_tools()`（可接受，接口预留的代价）

### ADR-010: Agent 运行时增强 — LLMConfig / streaming / 辩论上下文

| 字段 | 值 |
|------|-----|
| **日期** | 2026-06-07 |
| **状态** | ✅ 已采纳 |
| **影响范围** | `src/utils/llm.py`, `src/agents/base.py`, `src/core/protocol.py` |

**上下文**：

2026-06-07 全面项目审查发现 4 处骨骼级缺陷，必须在 Phase 0 修复，否则 Phase 1 辩论引擎将面临大规模接口重构：

| # | 缺陷 | 严重度 | 说明 |
|:--|:------|:------:|:-----|
| ① | LLM 参数硬编码 | S1 | `temperature=0.3`, `max_tokens=8192` 不可按 Agent 覆盖。辩论时各异需要不同人设（分析师 0.7 / 投票 0.1） |
| ② | 缺少 streaming 接口 | S2 | `ainvoke()` 返回 `str`，无 `astream() → AsyncIterator[str]`。Streaming UI 需要时需改 5 层代码 |
| ③ | AgentContext 无辩论槽位 | S1 | `input_data` 只有 `question`，MasterAgent 无法接收其他 Agent 的输出参与辩论 |
| ④ | LLM 实例缓存不支持多配置 | S2 | `get_llm()` 按 provider 名缓存，不同 temperature 的 Agent 拿到同一个实例，互相覆盖 |

**决策**：

#### 1. 引入 `LLMConfig` 数据类

```python
@dataclass
class LLMConfig:
    """LLM 调用配置 —— 定义一次调用的模型参数
    temperature / max_tokens / stream / reasoning_effort 均暴露给调用方。
    默认值与当前硬编码值一致，保证向后兼容。
    """
    temperature: float = 0.3
    max_tokens: int = 8192
    model: str | None = None          # None = 使用 provider 默认模型
    stream: bool = False
    reasoning_effort: str | None = None  # DeepSeek 推理强度标识
```

**接口变更**：

```python
# 当前签名（不修改，保持向后兼容）
async def ainvoke(self, prompt, system_prompt=None, ..., llm_config: LLMConfig | None = None) -> str

# 新增 streaming 接口
async def astream(self, prompt, system_prompt=None, ..., llm_config: LLMConfig | None = None) -> AsyncIterator[str]

# 缓存策略变更
def get_llm(self, config: LLMConfig | None = None, provider: str | None = None) -> BaseChatModel
```

#### 2. `AgentContext` 增加辩论槽位

```python
@dataclass
class AgentContext:
    session_id: str
    input_data: dict                      # 向后兼容：Question→Answer
    peer_outputs: list[AgentResult] = field(default_factory=list)  # 新增：辩论时其他 Agent 的输出
    current_round: int = 0                # 新增：辩论轮次
    target_audience: str = "user"         # "user" | "debate_group" | "master_vote"
    memory: dict = field(default_factory=dict)
    config: dict = field(default_factory=dict)
```

**向后兼容性**：
- `peer_outputs=[]`、`current_round=0`、`target_audience="user"` 为默认值
- 小智 / MasterAgent 现有 `input_data["question"]` 流程不感知这些字段
- 辩论引擎写入 `peer_outputs` 后, MasterAgent 可自然接收其他 Agent 的输出来回应

#### 3. LLM 实例缓存策略调整

```python
def _build_llm(self, config: LLMConfig, provider: str) -> BaseChatModel:
    """按 provider + config 创建 LLM 实例
    
    缓存策略：
    - LLMConfig 为默认值 → 按 provider 缓存（与当前行为一致）
    - LLMConfig 非默认值 → 每次都新建（不同 Agent 不同 temperature 不冲突）
    """
```

**理由**：

1. **Phase 0 修复成本最低** — 目前只有 2 个调用方（小智 + MasterAgent），改签名影响 20 行。Phase 1 有 10+ Agent 时改签名要动 20 个文件
2. **零向后兼容破坏** — `LLMConfig()` 默认值与当前硬编码值完全相同，所有现有调用传 `None` 则行为不变
3. **173 个现有测试零回归** — 默认路径测试不需要改
4. **AgentContext 辩论槽位不破坏现有请求路径** — `peer_outputs` 空列表相当于现有行为

**舍弃方案**：

| 方案 | 舍弃理由 |
|------|---------|
| Phase 1 再修，等辩论引擎写时一并改 | 辩论引擎 + 改 Agent 接口同时进行 = 双倍风险。现在拆开，每个变更可独立测试 |
| 不在 AgentContext 加辩论槽，辩论时传单独参数 | 接口碎片化，Agent.run() 要额外接收辩论参数，不利于 LangGraph StateGraph 统一管理 |
| streaming 等需要时再加 | `astream()` 不改变现有代码结构。现在加是加一个方法，以后加是要改所有相关的接口定义 |

**后果**：
- ✅ Agent 接口在 Q&A 和辩论两种模式中复用同一套签名
- ✅ 辩论引擎写的时候不需要回来改 Agent 的 `run()` 接口
- ✅ streaming UI 需要时 `llm_service.astream()` 可直接衔接任意前端框架
- ⚠️ `LLMConfig` 引入了新的配置对象，开发者需要理解（但 `None` = 默认值策略使复杂度降至最低）
