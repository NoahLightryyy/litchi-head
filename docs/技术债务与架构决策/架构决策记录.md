# 🏛️ 架构决策记录（Architecture Decision Records）

> 架构决策记录（ADR）用于记录每次重要技术选型的背景、方案对比和取舍理由。
> 让未来的开发者和 AI 都能理解"为什么要这样做"，避免重复讨论。

## 1. ADR 规范

### 1.1 什么时候写

以下情况**必须**记录 ADR：
- 引入新的外部依赖（库/框架/服务）
- 模块间的接口协议约定
- 数据存储方案选型
- 重大架构重构
- 舍弃了一个明显的替代方案

### 1.2 怎么写

每条 ADR 包含：

```markdown
## ADR-NNN: 标题

| 字段 | 值 |
|------|-----|
| 日期 | YYYY-MM-DD |
| 状态 | ✅ 已采纳 / 🔄 提议中 / ❌ 已废弃 / ⏪ 已替换 |
| 影响范围 | 受影响的模块/组件 |

**上下文**：
做这个决策的背景和驱动力。

**决策**：
最终选定的方案。

**理由**：
1. 选它的原因
2. ...

**舍弃方案**：
- 方案A：淘汰理由
- 方案B：淘汰理由

**后果**：
- 正面：...
- 负面：...（这就是引入的技术债务）
```

### 1.3 状态说明

| 状态 | 含义 |
|------|------|
| ✅ 已采纳 | 当前有效的决策 |
| 🔄 提议中 | 正在讨论，尚未决定 |
| ❌ 已废弃 | 明确放弃，不再考虑 |
| ⏪ 已替换 | 曾经有效，但已被新 ADR 取代 |

## 3. 决策记录

### ADR-001: Pydantic 作为全栈数据校验层

| 字段 | 值 |
|------|-----|
| **日期** | 2026-06-03 |
| **状态** | ✅ 已采纳 |
| **影响范围** | 全项目 |

**上下文**：
多 Agent 系统需要跨模块传递大量结构化数据，需要统一的序列化/校验方案。
同时，LLM 的结构化输出（`with_structured_output`）原生支持 Pydantic 模型。

**决策**：
所有模块间的数据传输统一使用 Pydantic `BaseModel`：
- 通信协议 `AgentMessage` — Pydantic
- Agent 输入/输出 — Pydantic
- 配置管理 — Pydantic Settings

**理由**：
1. 与 LLM `with_structured_output()` 天然对接
2. 零成本序列化/反序列化（`.model_dump()` / `.model_validate()`）
3. JSON Schema 导出能力方便调试和文档生成
4. Python 3.12 标准依赖，无额外引入

**舍弃方案**：
| 方案 | 舍弃理由 |
|------|---------|
| `dataclasses` 纯数据类 | 缺少校验和序列化能力 |
| `msgspec` | 性能更好但生态不如 Pydantic 与 LangChain 集成深 |
| Apache Avro / Protobuf | 太重，不适合 MVP 阶段 |

**后果**：
- ✅ 统一的数据契约层，减少模块间的沟通成本
- ✅ 调试友好（JSON Schema 可直接可视化）
- ⚠️ 性能不如 msgspec（但 MVP 阶段无感知）

### ADR-002: LangGraph 作为 Agent 编排框架

| 字段 | 值 |
|------|-----|
| **日期** | 2026-06-03 |
| **状态** | ✅ 已采纳 |
| **影响范围** | `src/core/`, `src/debate/` |

**上下文**：
需要框架来编排多个 Agent 的工作流，支持串行/并行/条件分支和状态共享。

**决策**：
使用 LangGraph 的 `StateGraph` 作为编排引擎。

**理由**：
1. 图结构天然适配 Agent 工作流（节点 = Agent，边 = 消息传递）
2. LangChain 生态原生支持
3. 内置并行节点和条件路由
4. 社区活跃，文档充足

**舍弃方案**：
| 方案 | 舍弃理由 |
|------|---------|
| AutoGen（微软） | 多 Agent 对话能力强但状态管理不够灵活 |
| CrewAI | 更上层封装但可控性不如 LangGraph |
| 自定义状态机 | 灵活但重复造轮子 |
| Temporal / Airflow | 太重，适合后端服务不适合 Agent 编排 |

**后果**：
- ✅ 得益于 LangGraph 生态，后续 LLM 集成省大量工作
- ⚠️ 版本更新快，需锁定版本

### ADR-003: akshare 作为首要 A 股数据源

| 字段 | 值 |
|------|-----|
| **日期** | 2026-06-03 |
| **状态** | ✅ 已采纳（MVP 阶段） |
| **影响范围** | `src/data/` |

**上下文**：
A 股数据源选择，需要免费、API 稳定、覆盖全面。

**决策**：
MVP 阶段使用 `akshare` 作为唯一数据源。

**理由**：
1. 免费开源，无需任何 Token 或注册
2. 覆盖 A 股行情、新闻、基本面、板块资金流
3. Python 原生库，pip install 即用
4. 社区维护，更新活跃

**限制**（需记录到技术债务）：
- 非官方数据源，宕机无 SLA 保障
- 请求频率受限
- 不支持 Level2 行情
- 同花顺账户对接仍需手工调研

**未来方案**：
Phase 2 考虑接入同花顺 iFinD / 聚宽数据。

### ADR-004: Streamlit 作为 MVP 前端框架

| 字段 | 值 |
|------|-----|
| **日期** | 2026-06-03 |
| **状态** | ✅ 已采纳（Phase 1 有效） |
| **影响范围** | `frontend/` |

**上下文**：
需要在前端展示辩论结果、账户信息，但前端开发资源有限。

**决策**：
Phase 1 使用 Streamlit 搭建 MVP 前端，Phase 2 再迁移到 React/Next.js。

**理由**：
1. 纯 Python，无需前端团队
2. 最快速度出可交互原型
3. `st.cache_data` 天然适配数据缓存场景
4. 部署简单（单服务器 + `streamlit run`）

**舍弃方案**：
| 方案 | 舍弃理由 |
|------|---------|
| React + Next.js | 学习成本高，Phase 0/1 资源不足 |
| Gradio | 更灵活但金融风格 UI 定制不如 Streamlit + Plotly |

**后果**：
- ✅ 非前端开发者也能快速搭界面
- ⚠️ Phase 2 重构为 React 时前端要重写（已知债务）

### ADR-005: 四组辩论架构 + 大师投票

| 字段 | 值 |
|------|-----|
| **日期** | 2026-06-03 |
| **状态** | ✅ 已采纳 |
| **影响范围** | `src/debate/`, `src/orchestrator/` |

**上下文**：
投资决策需要多角度分析，避免单一模型的偏见。

**决策**：
采用"四组风格辩论 → 大师投票 → 综合排序"的三层架构：

```
分析师(3人/组 × 4组) → 组内评审 → 跨组交叉质疑 → 大师投票 → 排序输出
```

**理由**：
1. 四种投资哲学覆盖不同市场环境
2. 组内质疑机制增加决策鲁棒性
3. 大师投票层模拟"投委会"决策流程
4. 每层可独立替换/升级

**舍弃方案**：
| 方案 | 舍弃理由 |
|------|---------|
| 单模型直接输出 | 缺乏辩论的鲁棒性 |
| 纯投票机制 | 缺少观点交锋过程，用户感知差 |

### ADR-006: 三层记忆架构

| 字段 | 值 |
|------|-----|
| **日期** | 2026-06-03 |
| **状态** | ✅ 已采纳 |
| **影响范围** | `src/memory/` |

**上下文**：
Agent 需要记忆来保证决策一致性，但 MVP 阶段不应过度工程化。

**决策**：
三层设计 + JSON 文件持久化起步：
- 工作记忆 — 当前 session 上下文
- 情景记忆 — 历史交易记录
- 反思机制 — LLM 驱动交易后复盘

**理由**：
1. 工作记忆保持对话上下文
2. 情景记忆记录历史交易供回溯
3. 反思机制是差异化功能（其他竞品少有）
4. JSON 文件起步降低开发复杂度

**舍弃方案**：
| 方案 | 舍弃理由 |
|------|---------|
| Milvus 向量数据库 | 需要部署服务，MVP 阶段过早 |
| Redis 缓存 | 功能过剩，JSON 文件已够用 |
| 无记忆系统 | Agent 每次"失忆"，决策质量差 |

### ADR-007: DeepSeek 作为主力 LLM

| 字段 | 值 |
|------|-----|
| **日期** | 2026-06-03 |
| **状态** | ✅ 已采纳（开发期） |
| **影响范围** | `src/utils/llm.py`, 成本 |

**上下文**：
API 成本控制对 MVP 阶段至关重要。

**决策**：
开发测试期默认使用 DeepSeek-Chat，保留 OpenAI 作为 fallback。
通过 `LLM_PROVIDER` 环境变量切换。

**成本对比**（单次辩论约 65k tokens）：

| 模型 | 单次成本 | 一个月开发（1000次） |
|------|---------|-------------------|
| DeepSeek-Chat | ~¥0.06 | ~¥60 |
| GPT-4o-mini | ~¥0.32 | ~¥320 |
| GPT-4o | ~¥1.92 | ~¥1,920 |
| Claude Sonnet 4 | ~¥1.07 | ~¥1,070 |

**理由**：
1. 成本仅为 GPT-4o 的 1/30
2. 中文金融场景表现足够
3. 保留 OpenAI API 作为 fallback

**后果**：
- ✅ 开发测试成本可控
- ⚠️ 最终上线可能需要切换到更强模型（需重新测试验证）
