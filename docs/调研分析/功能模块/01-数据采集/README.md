# 功能模块：数据采集

## 所属战线：金融数据聚合

> **战线格局**：OpenBB（全球统一数据层）vs AKShare（中国市场事实标准）vs 传统终端（Wind/彭博）
>
> [查看战线完整分析](../../行业开源方案对照知识库.md#战线-4金融数据聚合)

### 战线上谁在做什么

```
                    ┌── 统一数据层路线 ── OpenBB (64k⭐)
                    │    一次对接，随处调用，100+ Provider
                    │
  金融数据聚合 ────┼── 中国市场路线 ── AKShare
                    │    A股/期货/基金，0成本高质量，但接口不稳定
                    │
                    ├── 全流程内置路线 ── vnpy / Qlib
                    │    数据引擎内置于量化平台，不独立存在
                    │
                    └── 传统巨无霸 ── Wind / 彭博
                         付费，封闭，但数据深度无可替代
```

### 你的位置

你的 `src/data/` 目前走的是 **中国市场路线（AKShare）**，但架构上应向 **统一数据层路线（OpenBB）** 看齐——定义 Provider 抽象，让未来切换/增加数据源不改上层代码。

### 关键启示

| 来自谁 | 启示 |
|--------|------|
| **OpenBB** | Provider + Fetcher 三方法模式，这是数据层架构的标准答案 |
| **AKShare** | 接口上游不稳定，必须加适配器层做隔离 |
| **vnpy** | 事件驱动数据引擎适合未来实时行情场景 |
| **Qlib** | 数据量大时（800+股票×多年），二进制存储+多级缓存是必经之路 |

## 模块定义

多数据源行情/新闻/财务数据的获取、缓存、标准化。

**职责边界**：
- ✅ 多个数据源的统一接入（A股、港美股、加密货币等）
- ✅ 数据缓存与过期策略
- ✅ 原始数据 → Pydantic 模型的标准化转换
- ❌ 不负责数据的深度分析（那是辩论模块的事）
- ❌ 不负责因子计算（那是因子研究模块的事）

## 架构对照分析

### 你现在的架构（直接调 akshare）

```
┌─────────────────────────────────────────────┐
│  DataCollector                                │
│  ├─ get_realtime_quotes() → ak.stock_zh_a_spot_em()  │
│  ├─ get_kline() → ak.stock_zh_a_hist()               │
│  └─ get_news() → ak.stock_zh_a_news()                │
├─────────────────────────────────────────────┤
│  每个方法直接调 akshare，耦合紧，加数据源改接口   │
└─────────────────────────────────────────────┘
```

### OpenBB 的做法（你要借鉴的）

```
┌──────────────────────────────────────────────────────┐
│  接口层                                                │
│  obb.equity.price.historical(symbol="000001")         │
├──────────────────────────────────────────────────────┤
│  Router / CommandRunner（命令路由）                   │
├──────────────────────────────────────────────────────┤
│  Provider 系统（每个数据源独立实现）                   │
│  ├─ AkShareProvider ──→ 调 akshare 函数           │
│  ├─ TuShareProvider ──→ 调 tushare API            │
│  ├─ YahooFinanceProvider ──→ 调 yfinance           │
│  └─ 未来自定义数据源 ──→ 只要实现 Fetcher 接口     │
├──────────────────────────────────────────────────────┤
│  每个 Provider 的 Fetcher：                            │
│  1. transform_query() — 标准参数 → 源特有参数     │
│  2. extract_data() — 实际 API 调用               │
│  3. transform_data() — 原始数据 → Pydantic 模型  │
└──────────────────────────────────────────────────────┘
```

### vnpy 的事件驱动方式（未来实时行情）

```
┌──────────────────────────────────────┐
│  EventEngine（事件引擎）               │
│  ├─ 行情事件 → 推送给订阅者            │
│  ├─ 订单事件 → 推送给策略模块          │
│  └─ 成交事件 → 推送给风控/记账模块    │
├──────────────────────────────────────┤
│  不主动"拉"数据，而是数据到了"推"给订阅者  │
│  适合实时行情场景（WebSocket/CTP推送）  │
└──────────────────────────────────────┘
```

### Qlib 的数据加速方案（未来大数据量）

```
原始CSV → dump_bin.py → 二进制文件 → MemCache(LRU) → DataSet
                              ↑
             多级缓存：内存/Redis/文件，比MySQL快50倍
```

## 深挖方向建议（2026-06-11 更新）

### 🥇 简报分区输出 C1（数据模块 ← 辩论模块需求）

**来源**：TradingAgents 信息隔离设计的"适配版本" | **工作量**：小 | **影响范围**：`data/collector.py`

`format_market_brief()` 目前输出全量市场简报。建议按区块分区输出，让大师看到结构化的数据分层：

```
----- 行情层 -----
(实时报价、涨跌幅、换手率)

----- 新闻层 ----
(最新相关新闻摘要)

----- 情绪层 -----
(舆情指标、社交媒体热度)

----- 基本面层 ----
(PE/PB、营收、利润等)
```

不隔离数据（大师仍然看到全貌），但显式分区让大师的"人格视角"更容易聚焦。

### 🥈 架构层：Provider 抽象层
设计 `DataProvider` / `Fetcher` 接口，实现 `AkShareProvider` 做验证。目标：上层代码不关心数据源哪来的。
