# 🔁 第9层·用户经验反馈闭环 — 完整实施计划

> **一句话**：记录 AI 推荐了什么 → 你做了什么 → 最后赚了亏了 → 系统从你的操作中学习 → 下次决策更懂你。
>
> **三权分立定位**：根据 [DESIGN_PHILOSOPHY.md](../00-overview/DESIGN_PHILOSOPHY.md)，本条链路贯彻"公式负责调整、镜子负责展示、人负责拍板"。
> - **公式调整**：ResultCallbackEngine 自动根据结果参数回调相关策略（RC 系列）
> - **镜子展示**：RetroBoard + BehaviorComparisonReport 让你看清自己的模式
> - **人拍板**：系统永不替你做决定，只提供结构化历史参考

## 1. 产品哲学 — 为什么这是一个闭环

当前系统的最大问题不是"AI 分析不准"，而是**AI 分析完就结束了，没人知道用户的真实操作和结果**：

```
你问："茅台能买吗？"
系统输出：买入 (置信度 72%)
  → 你买了还是没买？ → 不知道
  → 你买了之后涨了还是跌了？ → 不知道
  → 你的操作逻辑是什么？ → 不知道
  → 下次系统怎么改进？ → 改进不了
```

这就是你说的"结果参数回调相关策略"——结果到了，全链路参数都不响应。

**第 9 层的存在意义**：不让任何一次 AI 输出和用户操作变成"一次性的"。每一条记录都在让系统变得更了解你。

## 3. 8 个组件详解

### 组件 1：AI 推荐记录（已有 ✅）

| 属性 | 值 |
|:-----|:----|
| 代码位置 | `src/debate/models.py` — `DebateResult` + `VoteSummary` |
| 存储位置 | `data/memory/episodic/debate/{stock_code}.jsonl` |
| 生产方 | `DebateOrchestrator._save_decision_to_memory()` |
| 状态 | **已有** — 每次辩论自动记录 |

**缺口**：记录中存在 `consensus` / `weighted_score` / `confidence`，但没有"每个大师对当前股票的具体评分"的可追溯记录。M3 TrustTracker 本应做这件事，但 `record_outcome()` 从未被调用。

### 组件 2：用户操作采集（RC-003 待实现 🆕）

| 属性 | 值 |
|:-----|:----|
| 代码位置 | `src/callback/callbacks/ub_track.py` — `InvestmentDecision` + `UserBehaviorStore` |
| 存储位置 | `data/user_profiles/{user_id}/decisions.jsonl` |
| 触发方式 | 用户在前端点"买入/卖出/关注/忽略" |
| 依赖 | RC-001 核心引擎 + `POST /api/user/action` 端点 |
| 状态 | **规划完成·待实施** |

**关键设计**：每个用户操作必须同时记录以下信息才能形成闭环：

```json
{
  "ticker": "600519",
  "direction": "buy",
  "ai_recommendation": "buy",
  "ai_recommendation_confidence": 0.82,
  "quantity": 100,
  "price": 168.50,
  "stated_reason": "感觉跌到位了",
  "reason_category": "情绪驱动",
  "session_id": "debate_xxx"
}
```

**为什么这个组件是闭环的基础**：没有用户操作记录，后面的"对比"和"学习"全是空话。

### 组件 3：实际盈亏追踪（RC-003 扩展 🆕）

| 属性 | 值 |
|:-----|:----|
| 触发方式 | 用户卖出时自动计算盈亏；或后端定时扫描未了结交易并计算浮动盈亏 |
| 数据 | `InvestmentDecision.actual_outcome` / `actual_return_pct` / `holding_days` |
| 依赖 | 用户操作采集 → 卖出/了结事件 |
| 状态 | **待设计** |

**注意**：这不是回测的模拟盈亏，是**用户的真实操作盈亏**。只有用户自己知道什么时候以什么价格买了多少——所以这部分数据必须从前端主动上报，后端无法自动获取。

### 组件 4：公式层 — ResultCallbackEngine（RC-001~006）

| 组件 | 功能 | 状态 |
|:-----|:-----|:----:|
| RC-001 核心引擎 | 事件分发器 + 注册表 + 存储层 | 🟦 规划通过 |
| RC-002 M3-EXT | 按板块的大师信任度校准 + sector 字段 | 🟦 规划通过 |
| RC-004 RP-TUNE | 风险参数根据回测结果自适应 | 🟦 规划通过 |
| RC-005 CALIBRATE | 置信度 Brier score 过高时注入校准乘数 | 🟦 规划通过 |
| RC-006 STRAT-ROUTE | 不同市场条件下的大师路由 | 🟦 规划通过 |

**与闭环的关系**：这是第 9 层的"自动执行器"。用户不需要知道它的存在——它沉默地在后台接收结果、调整参数、改进下次辩论。

### 组件 5：RetroBoard 交易复盘看板（R4 🆕）

| 属性 | 值 |
|:-----|:----|
| 用途 | 让用户一目了然地看到 AI 推荐了什么、自己做了什么、结果如何 |
| 展示形式 | 表格（三列对比）+ 聚合卡片（准确率/胜率/最佳 Agent） |
| 后端 | `GET /api/retro/` — 按日期/股票/结果筛选 |
| 前端 | 新页面（`/retro`）或嵌入个股页 |
| 依赖 | 组件1（AI推荐）+ 组件2（用户操作）+ 组件3（实际盈亏） |
| 状态 | **ROADMAP 有设计·待实现** |

**为什么不是简单的表**：RetroBoard 的核心价值不是展示数据，是**让你一眼看出自己的行为模式**。

```
┌──────────┬─────────┬──────────┬───────┬──────┬────────┬──────────┬──────────┐
│ 日期     │ 代码    │ AI 方向  │ 置信度 │ 你操作 │ 盈亏%  │ 正确?    │ 偏差     │
├──────────┼─────────┼──────────┼───────┼──────┼────────┼──────────┼──────────┤
│ 06-17   │ 茅台   │ 买入     │ 82%   │ 买入 │ +3.2%  │ ✅ AI 对  │ —        │
│ 06-17   │ 平安   │ 卖出     │ 65%   │ 持有 │ -1.5%  │ ⚠️ 你没听  │ 过度自信  │
│ 06-16   │ 宁德   │ 买入     │ 91%   │ 买入 │ -2.1%  │ ❌ AI 错  │ 市场突发  │
└──────────┴─────────┴──────────┴───────┴──────┴────────┴──────────┴──────────┘

聚合卡片：
┌────────────┬────────┬────────┬────────┐
│ AI 准确率  │ 你跟AI  │ 你逆AI  │ 最佳    │
│ 65%       │ 的胜率  │ 的胜率  │ Agent   │
│           │ 72%    │ 33%    │ 巴菲特  │
└────────────┴────────┴────────┴────────┘
```

### 组件 6：行为对比报告 — 镜子 Agent（DP-006 🆕）

| 属性 | 值 |
|:-----|:----|
| 用途 | 在你做决策前，展示"你当前的操作和历史相似模式"的对比 |
| 输出 | `BehaviorComparisonReport` — 当前操作摘要 + 相似历史 + 行为模式总结 |
| 触发时机 | 决策前（可选 toggle）+ 闭市后复盘 |
| 三段式解锁 | 1-9次仅记录 → 10+次出对比报告 → 30+次可参与辩论 |
| 依赖 | 组件2（用户操作）+ 组件3（实际盈亏）→ 数据积累 |
| 状态 | **完全设计文档在 `docs/99-archive/` · 待实现** |

**和 RetroBoard 的区别**：
- RetroBoard = **事后看**：所有历史汇总在一张表，需要你主动点开看
- 镜子 Agent = **事前提醒**：在你做决策弹出的提示，"你之前这样也亏过"

### 组件 7：M2 反思注入（已有骨架 ✅）

| 属性 | 值 |
|:-----|:----|
| 代码位置 | `src/debate/reflection.py` — `_format_reflection_context()` |
| 存储位置 | `data/memory/reflective/debate/` |
| 当前状态 | **骨架就绪**：能从 MemoryStore 读取反思并注入大师 prompt |
| 缺口 | 反思数据来源只有"AI 预测 vs 市场结果"，没有"用户操作偏差"维度 |

**升级方向**：当用户的操作模式和 AI 推荐出现系统性偏差（如总在 AI 说"持有"时提前卖出），M2 反思的输入应该加入用户行为数据，生成更个性的反思文本："你在白酒板块有抄底倾向（3/3次），本次 AI 也推荐买入，但你上次在科技股抄底亏损 5%，需要特别注意。"

### 组件 8：用户行为存储（RC-003 存储层 🆕）

| 属性 | 值 |
|:-----|:----|
| 位置 | `data/user_profiles/{user_id}/decisions.jsonl` + `profile.json` |
| 格式 | JSONL 决策流水 + JSON 行为画像汇总 |
| 隔离 | 与金融知识库完全隔离（用户隐私） |
| 检索 | `UserBehaviorStore.get_decisions(user_id, limit)` + `get_profile(user_id)` |
| 状态 | **规划完成·待实施** |

## 6. 与现有系统的关系

### 不走回头路的设计原则

| 原则 | 说明 |
|:-----|:------|
| **用户数据隔离** | `data/user_profiles/` 独立目录，不与金融知识库混存 |
| **默认不激活** | 用户需要主动开启行为记录（隐私合规） |
| **输出不说"你应该"** | 镜子 Agent 只展示模式，让用户自己判断 |
| **公式不经过 LLM** | RC 系列回调全部确定性计算（阈值/公式），不调用 LLM，防止自我确认偏误 |
| **冷启动保护** | 数据 < 10 条不输出对比报告，宁可不说话也不说没根据的话 |

### 与 ADR 的关系

| ADR | 关联 |
|:----|:------|
| ADR-006 三层记忆 | 用户行为记忆 + 用户画像 = 第三层"长期记忆" |
| ADR-008 Agent 通用化 | 镜子 Agent 继承 BaseAgent，输出 BehaviorComparisonReport |
| ADR-009 MCP 工具 | 镜子 Agent 通过 MCP 查询用户行为存储 |
