## 1. 核心原则

```
┌─────────────────────────────────────────────┐
│  每次 AI 会话 = 一次完整的工作单元           │
│                                             │
│  1️⃣ 启动时：理解上下文 + 加载最新状态        │
│  2️⃣ 工作中：代码 + 文档 + 测试同步推进       │
│  3️⃣ 结束时：更新日志 + 更新债务 + 提交记录    │
└─────────────────────────────────────────────┘
```

### 1.1 四同步原则

任何变更必须同步更新四样东西：

| # | 要素 | 说明 |
|---|------|------|
| 🖥️ | **代码** | 功能实现 |
| 🧪 | **测试** | 新功能必须有测试覆盖（目标 80%+） |
| 📝 | **文档** | README / 设计文档 / ADR |
| 🐛 | **债务日志** | 新增债务或更新状态 |

> 只改代码不改文档 = 留下新债务。
> 只改文档不改代码 = 空谈。
> 改了代码不补测试 = 债务正在累积。

### 1.2 优先序原则

处理任务时按此优先级：

```
安全/数据完整性 → 阻塞项(S0) → 核心功能 → 可维护性 → 整洁优化
      🥇             🥈           🥉          🎯         💡
```

## 2. 会话启动标准流程

每次 AI 开始新会话时，按此顺序加载状态：

### 2.1 第 1 步：检查项目状态

```markdown
- [ ] 阅读 CLAUDE.md（若存在）获取项目级指令
- [ ] 阅读 `docs/流程规范/AI自动化工作流程.md`（即本文档）
- [ ] **查看 `docs/计划/README.md` 项目看板 — 全局进度总览**
- [ ] 检查 `docs/ai-work-logs/README.md` 了解最近工作
- [ ] 检查 `docs/技术债务与架构决策/技术债务日志.md` 最新债务
- [ ] 检查当前 git 状态（是否有未提交变更）
- [ ] 确认 `.env` 配置是否完整（API Key 等）
- [ ] 确认 `.gitignore` 包含 `.env`（防误上传）
- [ ] 确认 `.env` 未被 Git 追踪（`git check-ignore .env`）
- [ ] 根据任务类型启用对应子代理（详见 §3.3）
```

### 2.2 第 2 步：理解上下文

```markdown
- [ ] 明确本次会话的目标任务
- [ ] 确认依赖的模块是否已经就绪
- [ ] 如果涉及跨模块变更，先确认接口契约（Pydantic 模型）
- [ ] 如果涉及技术选型，先查阅 ADR 避免重复决策
```

## 3. 日常开发流程

### 3.1 功能开发标准步骤

```
需求确认 → 接口设计 → 实现 → 自测 → 文档 → 债务检查 → 代码审查 → 提交
   📋       📐       🔨    🧪    📝      🐛        🧐        📤
```

1. **需求确认** — 理解要做什么，明确验收标准
2. **接口设计** — 先定义 Pydantic 模型和函数签名（契约优先）
3. **实现** — 编写功能代码
4. **自测** — 至少跑一次手动验证，有测试则跑测试
5. **文档** — 更新相关 `.md` 文档
6. **债务检查** — 本次实现是否引入了新的债务？→ 登记
7. **代码审查** — 使用 `code-reviewer` 代理审查变更；若涉及安全敏感代码先使用 `security-reviewer` 代理
8. **提交** — 规范的 git commit message

### 3.2 代码质量门槛

**每次提交前必须通过**以下全部检查（一键运行 `make check`）：

| 检查项 | 命令 | 通过标准 |
|--------|------|---------|
| Ruff 代码风格 | `make lint`（`ruff check .`） | 0 errors |
| Pyright 类型检查 | `make type`（`pyright src/`） | 0 errors |
| Pytest 测试 | `make test`（`pytest -v --tb=short`） | 所有测试通过 |

> **测试要求**：每项变更都应有对应的测试覆盖。新功能代码必须添加测试，bug 修复必须追加回归测试。

### 3.3 Agent 编排指引

根据任务类型，按需启用子代理配合执行：

| 场景 | 推荐代理 | 用途 |
|------|---------|------|
| 🏗️ 复杂功能/重构 | `planner` | 制定实现计划、分解任务 |
| 🔧 架构决策 | `architect` | 系统设计、方案对比 |
| 🧪 新功能 / Bug 修复 | `tdd-guide` | 先写测试（RED）→ 实现（GREEN）→ 重构（IMPROVE） |
| 🧐 代码刚写完 | `code-reviewer` | 代码质量审查（CRITICAL/HIGH 必须修复） |
| 🔒 安全敏感代码 | `security-reviewer` | 检查 OWASP Top 10、密钥泄露、注入风险 |
| 🐛 构建失败 | `build-error-resolver` | 分析错误日志、增量修复 |
| 📖 文档更新 | `doc-updater` | 更新 README / ADR / 工作日志 |

> 对于复杂任务，可同时启动多个子代理并行工作（如安全审查 + 代码审查同时进行）。

### Agent 不可用时降级方案

Agent 依赖外部 API，偶发不可用时有降级预案：

| 场景 | 默认 Agent | 降级方案 |
|------|-----------|---------|
| 🔍 代码审查 | `code-reviewer` | 手动审查，收紧标准（CRITICAL/HIGH 必修复） |
| 🏗️ 规划 | `planner` | 改用`EnterPlanMode`模式，手动分解任务 |
| 🧪 TDD | `tdd-guide` | 按照文档 §3.1 标准步骤人工执行 RED→GREEN→IMPROVE |
| 🔒 安全审查 | `security-reviewer` | 运行 `bandit -r src/` 后手动审查结果 |
| 📖 文档 | `doc-updater` | 按模板手工创建/更新文件 |

> 降级不意味着降低质量门槛——审查标准不变，只是从 AI 助审变为人工审核。

### 3.4 集成测试规范

真实外部依赖（LLM API、数据库、第三方服务）的集成测试需遵守：

| 规则 | 说明 |
|------|------|
| `@pytest.mark.integration` | 所有调用真实 API 的测试必须标记此 marker |
| `@skip_no_key` 装饰器 | 环境变量未配置时自动跳过，保证 CI 通过 |
| `tmp_path` 隔离 | 测试使用临时目录，不写真实数据目录 |
| 独立文件 | 集成测试与单元测试分文件存放（`test_integration_*.py`） |
| 无 `src/` 改动 | 新增集成测试原则上不应修改业务代码 |

**skip_no_key 标准实现**：

```python
def _has_api_key() -> bool:
    try:
        from src.utils.config import settings
        return bool(settings.deepseek_api_key)
    except Exception:
        return False

skip_no_key = pytest.mark.skipif(
    not _has_api_key(),
    reason="需要 API Key（在 .env 中配置）",
)
```

> 集成测试不参与覆盖率统计，但须保证无 API Key 时零失败。

## 4. 每次审视必查清单

> 每次 AI 审视项目时，必须系统性地检查以下内容。
> 这是发现债务的主要途径。

### 4.1 架构层面

- [ ] 是否有模块缺少必要的抽象层？
- [ ] 模块间接口是否稳定？是否需要新的 ADR？
- [ ] 现有 ADR 是否仍然有效？是否有被新情况推翻的？
- [ ] 依赖项是否有版本冲突或过时的？

### 4.2 代码层面

- [ ] 是否有未处理的 `# TODO` / `# FIXME` / `# HACK` 注释？
- [ ] 是否有函数签名缺少类型注解？
- [ ] 是否有 `dict = None` 这类 Pyright 不通过的写法？
- [ ] 是否有硬编码的配置值（API Key、URL、价格）？
- [ ] 是否有重复代码段可以抽取？
- [ ] 是否有 import 未使用或排序混乱？

### 4.3 测试层面

- [ ] 是否有模块缺少对应的测试文件？
- [ ] 新的功能代码是否可测试？是否需要 Mock 工具？
- [ ] CI 流水线是否正常通过？

### 4.4 文档层面

- [ ] 新增功能是否有对应的文档说明？
- [ ] 配置项变化是否更新了 `.env.example`？
- [ ] AI 工作日志是否需要更新？
- [ ] 交接文档（`docs/流程规范/AI会话交接文档.md`）是否需要同步更新？

### 4.5 安全层面

- [ ] 是否有敏感信息可能泄露（API Key、密码）？
- [ ] 是否有未鉴权的操作？
- [ ] 日志中是否会打印密钥等敏感信息？

## 5. 会话结束标准流程

### 5.1 结束 checklist

```markdown
## 会话结束 checklist

### ☑️ 文档更新
- [ ] AI 工作日志已更新（`docs/ai-work-logs/YYYY/MM/DD/YYYY-MM-DD.md`）
- [ ] 日志 README 索引已更新（如有新增文件）
- [ ] 债务日志已更新（如有新增/变更债务）
- [ ] 相关的设计文档已同步更新

### ☑️ 代码质量
- [ ] Ruff 检查通过
- [ ] Pyright 检查通过
- [ ] Git 工作区干净或有合理的未提交变更
- [ ] `git status` 确认 ahead commits 已标记「待推送」

### ☑️ 任务闭环
- [ ] 已完成本次目标任务
- [ ] 未完成的部分记录在日志的「待办事项」中
- [ ] 阻塞项已标记清楚（🔴）
```

### 5.2 日志模板

每次会话结束必须创建/更新日志文件 `docs/ai-work-logs/YYYY/MM/DD/YYYY-MM-DD.md`：

```markdown
# AI 工作日志 — YYYY-MM-DD

## 本次工作

### 1. 标题

**涉及文件**：
- ⭐ `path/to/new.file` — 新增
- 🔧 `path/to/modified.file` — 修改

**内容**：
（做了什么、为什么）

### 2. ...

## 当前项目状态

（简要描述完成进度，如"Phase 0 基建中，LLM 层已完成"）

## 债务变更

| ID | 操作 | 说明 |
|----|------|------|
| TD-XXX | 新增/修复/更新 | ... |

## 待办事项
- [ ] 下次要做的事
```

## 6. 债务管理流程

### 6.1 发现即登记

审视中发现的任何债务，**立即登记**，三步完成：

```
① 复制模板（债务新增模板.md）→ ② 填写四维分类 → ③ 追加到债务日志
```

### 6.2 每天更新规则

| 时段 | 动作 | 说明 |
|------|------|------|
| 🟢 **无新债务** | 日志中备注"本次未发现新债务" | 保持习惯 |
| 🟡 **有新债务** | 追加到 §5 + 更新 §6 日期索引 | 按模板填写 |
| 🟠 **状态变更** | 更新已有债务的状态（如→🔧修复中） | 联动 AI 工作日志 |
| 🔴 **关闭债务** | 标记 ✅ 已关闭 + 记录实际工时 | 更新仪表盘数据 |

### 6.3 利息定期复审

| 状态 | 复审频率 | 动作 |
|------|---------|------|
| 📖 已接受 | 每月 | 检查利息是否超标，需要重新排期？ |
| 🆕 待评估 | 每周 | 积压太多要批量评估 |
| 📋 已确认 | 每 Sprint | 确认排期优先级 |

## 7. 日志更新规范

### 7.1 AI 工作日志

- **文件名**：`docs/ai-work-logs/YYYY/MM/DD/YYYY-MM-DD.md`
- **单日多次对话**：`YYYY-MM-DD-N.md`（N=1,2,3...）
- **每次会话都写**：即使只做了小改动
- **内容要求**：记录做了什么、改了什么、为什么、下一步建议

### 7.2 债务日志

- **按日期分组**：§6 日期索引按发现日分组展示
- **按状态管理**：§5 按生命周期状态分组（待评估/已确认/修复中等）
- **新债追加**：§5 对应分组 + §6 对应日期
- **状态迁移**：债务从"已确认"→"修复中"时，更新 §5 位置

### 7.3 日志间交叉引用

```markdown
AI 工作日志  ←引用→  债务日志（记录新增/变更了哪些债务）
债务日志     ←引用→  AI 工作日志（"修复方向参见某日日志"）
```

## 8. 突发情况处理

### 8.1 遇到阻塞

```markdown
1. 记录阻塞原因到 AI 工作日志
2. 如果是代码问题 → 评估是否要新增债务
3. 如果是外部依赖（API 不可用等）→ 记录等待
4. 明确告知用户阻塞了什么，建议下一步
```

### 8.2 上下文耗尽

```markdown
1. 执行会话结束 checklist（§5.1）
2. 确保 AI 工作日志完整
3. 确保债务日志已更新
4. 在日志末尾标记「上下文耗尽，续接」
```

### 8.3 发现之前的决策有误

```markdown
1. 更新相关 ADR（标记为 ⏪ 已替换）
2. 创建新的 ADR 记录新的决策
3. 在 AI 工作日志中说明变更理由
4. 更新受影响的代码和文档
```
