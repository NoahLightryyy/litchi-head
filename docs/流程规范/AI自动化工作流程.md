## 1. 核心原则

```
┌─────────────────────────────────────────────┐
│  每次 AI 会话 = 一次完整的工作单元           │
│                                             │
│  1️⃣ 启动时：理解上下文 + 加载最新状态        │
│  2️⃣ 工作中：代码 + 文档 + 测试同步推进       │
│  3️⃣ 结束时：更新日志 + 更新债务 + 提交记录    │
└─────────────────────────────────────────────┘
```

### 1.1 四同步原则

任何变更必须同步更新四样东西：

| # | 要素 | 说明 |
|---|------|------|
| 🖥️ | **代码** | 功能实现 |
| 🧪 | **测试** | 新功能必须有测试覆盖（目标 80%+） |
| 📝 | **文档** | README / 设计文档 / ADR |
| 🐛 | **债务日志** | 新增债务或更新状态 |

> 只改代码不改文档 = 留下新债务。
> 只改文档不改代码 = 空谈。
> 改了代码不补测试 = 债务正在累积。

### 1.2 优先序原则

处理任务时按此优先级：

```
安全/数据完整性 → 阻塞项(S0) → 核心功能 → 可维护性 → 整洁优化
      🥇             🥈           🥉          🎯         💡
```

### 1.3 重大决策调研原则

遇到涉及架构选型、模块新增、技术方案对比的重大决策时，遵循以下流程：

```markdown
### 重大决策三步法
1. 🔍 **先调研** — 搜索市场中同类竞品/成熟框架的做法（至少 2-3 个来源）
2. 📊 **对照分析** — 提炼各方案的核心设计 + 适用场景 + 优缺点
3. 🧑‍💻 **由人决策** — 将调研结论（含推荐方案）呈现给开发者，由人做最终选择

### 什么时候必须调研
- 新增一个完整模块（数据库、记忆系统、缓存层等）
- 涉及存储/基础设施选型（存什么、怎么存）
- 系统间的集成模式（同步/异步/事件驱动）
- 任何改动后难以回退的决策

### 原则
- 调研不意味着照搬——根据本项目的实际情况（技术栈、阶段、团队）做适配
- 竞品做的好的地方要借鉴，做的不好的要避开
- 调研结果必须记录到 ADR 的「舍弃方案」或「备选方案」部分
- 最终决策权在开发者，AI 提供分析但不代为决断
```

## 2. 会话启动标准流程

每次 AI 开始新会话时，按此顺序加载状态：

### 2.1 第 1 步：检查项目状态

> **上下文优化**：核心信息（身份卡/ADR/当前状态）已由 Memory 系统自动注入，
> 无需再逐份完整读取 CLAUDE.md、债务日志和设计文档。
> 基于约定：memory/ 中的 3 条 key facts 永远在线。

```markdown
### 自动注入（不占手动读取）
- [ 自动] project-identity → 技术栈、红线、定位（memory/）
- [ 自动] architecture-decisions → ADR 要点（memory/）
- [ 自动] current-state → 活跃任务、模块完成度（memory/）

### 快速同步（启动时读取，~3-5K token）
- [ ] 阅读 `docs/流程规范/AI会话交接文档.md §2（当前状态）+ §5（下一步）`
- [ ] 阅读 `docs/ai-work-logs/README.md` 了解最近工作索引
- [ ] 阅读最新工作日志（前次会话做了什么）

### 快速检查
- [ ] 检查当前 git 状态（是否有未提交变更）
- [ ] 确认 `.env` 配置是否完整（API Key 等）
- [ ] 确认 `.gitignore` 包含 `.env`（防误上传）
- [ ] 确认 `.env` 未被 Git 追踪（`git check-ignore .env`）
- [ ] 根据任务类型启用对应子代理（详见 §3.3）

### 按需加载（用到时才读）
- [按需] `docs/技术债务与架构决策/技术债务日志.md` — 要修债务时再读
- [按需] `docs/计划/README.md` 项目看板 — 要查全局进度时
- [按需] `docs/架构设计/` 设计文档 — 要修改架构时
- [按需] 本工作流程文档 §3~§8 — 进行开发/审视/终止等具体流程时
```

### 2.2 第 2 步：理解上下文

```markdown
- [ ] 明确本次会话的目标任务
- [ ] 确认依赖的模块是否已经就绪
- [ ] 如果涉及跨模块变更，先确认接口契约（Pydantic 模型）
- [ ] 如果涉及技术选型，先查阅 ADR 避免重复决策
```

## 3. 日常开发流程

### 3.1 功能开发标准步骤

```
需求确认 → 接口设计 → 实现 → 自测 → 文档 → 债务检查 → 代码审查 → 提交
   📋       📐       🔨    🧪    📝      🐛        🧐        📤
```

1. **需求确认** — 理解要做什么，明确验收标准
2. **接口设计** — 先定义 Pydantic 模型和函数签名（契约优先）
3. **实现** — 编写功能代码
4. **自测** — 至少跑一次手动验证，有测试则跑测试
5. **文档** — 更新相关 `.md` 文档
6. **债务检查** — 本次实现是否引入了新的债务？→ 登记
7. **代码审查** — 使用 `code-reviewer` 代理审查变更；若涉及安全敏感代码先使用 `security-reviewer` 代理
8. **提交** — 规范的 git commit message

### 3.2 代码质量门槛

**每次提交前必须通过**以下全部检查（一键运行 `make check`）：

| 检查项 | 命令 | 通过标准 |
|--------|------|---------|
| Ruff 代码风格 | `make lint`（`ruff check .`） | 0 errors |
| Pyright 类型检查 | `make type`（`pyright src/`） | 0 errors |
| Pytest 测试 | `make test`（`pytest -v --tb=short`） | 所有测试通过 |

> **测试要求**：每项变更都应有对应的测试覆盖。新功能代码必须添加测试，bug 修复必须追加回归测试。

### 3.3 Agent 编排指引

根据任务类型，按需启用子代理配合执行：

| 场景 | 推荐代理 | 用途 |
|------|---------|------|
| 🏗️ 复杂功能/重构 | `planner` | 制定实现计划、分解任务 |
| 🔧 架构决策 | `architect` | 系统设计、方案对比 |
| 🧪 新功能 / Bug 修复 | `tdd-guide` | 先写测试（RED）→ 实现（GREEN）→ 重构（IMPROVE） |
| 🧐 代码刚写完 | `code-reviewer` | 代码质量审查（CRITICAL/HIGH 必须修复） |
| 🔒 安全敏感代码 | `security-reviewer` | 检查 OWASP Top 10、密钥泄露、注入风险 |
| 🔬 全面项目审视 | `architect` + `code-reviewer` | 多维度系统性审查（架构+代码+测试+文档+安全） |
| 🎯 针对性审视 | `code-reviewer` / `security-reviewer` | 聚焦特定模块的深度审查 |
| 🐛 构建失败 | `build-error-resolver` | 分析错误日志、增量修复 |
| 📖 文档更新 | `doc-updater` | 更新 README / ADR / 工作日志 |

> 对于复杂任务，可同时启动多个子代理并行工作（如安全审查 + 代码审查同时进行）。

### Agent 不可用时降级方案

Agent 依赖外部 API，偶发不可用时有降级预案：

| 场景 | 默认 Agent | 降级方案 |
|------|-----------|---------|
| 🔍 代码审查 | `code-reviewer` | 手动审查，收紧标准（CRITICAL/HIGH 必修复） |
| 🏗️ 规划 | `planner` | 改用`EnterPlanMode`模式，手动分解任务 |
| 🧪 TDD | `tdd-guide` | 按照文档 §3.1 标准步骤人工执行 RED→GREEN→IMPROVE |
| 🔒 安全审查 | `security-reviewer` | 运行 `bandit -r src/` 后手动审查结果 |
| 🔬 审视 | `architect` + `code-reviewer` | 按 §4 审视清单逐项手动检查，按 §3.5 审视工作流执行 |
| 📖 文档 | `doc-updater` | 按模板手工创建/更新文件 |

> 降级不意味着降低质量门槛——审查标准不变，只是从 AI 助审变为人工审核。

### 3.4 集成测试规范

真实外部依赖（LLM API、数据库、第三方服务）的集成测试需遵守：

| 规则 | 说明 |
|------|------|
| `@pytest.mark.integration` | 所有调用真实 API 的测试必须标记此 marker |
| `@skip_no_key` 装饰器 | 环境变量未配置时自动跳过，保证 CI 通过 |
| `tmp_path` 隔离 | 测试使用临时目录，不写真实数据目录 |
| 独立文件 | 集成测试与单元测试分文件存放（`test_integration_*.py`） |
| 无 `src/` 改动 | 新增集成测试原则上不应修改业务代码 |

**skip_no_key 标准实现**：

```python
def _has_api_key() -> bool:
    try:
        from src.utils.config import settings
        return bool(settings.deepseek_api_key)
    except Exception:
        return False

skip_no_key = pytest.mark.skipif(
    not _has_api_key(),
    reason="需要 API Key（在 .env 中配置）",
)
```

> 集成测试不参与覆盖率统计，但须保证无 API Key 时零失败。

### 3.5 审视/审核工作流

> 审视是一个独立的工作类型，与功能开发（§3.1）并列。
> 不是所有审视都需要相同的工作量，但所有审视都必须完成其级别对应的核心步骤。

#### 审视级别定义

| 级别 | 适用场景 | 预估耗时 | 输出要求 |
|:----:|:---------|:--------:|:---------|
| 🔬 **全面审视** | 项目里程碑、阶段切换、长期未审视 | 30-60min | 完整审视报告 + 全部文档同步 |
| 🎯 **针对性审视** | 聚焦特定模块/债务/风险 | 10-20min | 聚焦报告 + 相关文档同步 |
| ⚡ **快速审视** | 日常状态检查、微小变更审查 | 5-10min | 仅更新工作日志 |

#### 审视标准步骤（以全面审视为准）

```
审视启动 → 系统检查 → 输出结论 → 用户确认 → 落笔同步 → 代码审查 → 提交与推送
   📋       🔍        📝        ✅        ✍️        🧐        📤
```

**1. 审视启动** — 明确审视范围和级别
- 明确本次审视的目标（全貌？模块？债务？安全？）
- 回顾上次审视的结论和未完成事项

**2. 系统检查** — 按审视级别执行对应的 §4 审视清单
- 全面审视：执行全部 §4.1~4.5 五个层面
- 针对性审视：仅执行相关层面
- 快速审视：仅执行 §4.2 代码层面和 §4.3 测试层面

**3. 输出审视结论**
- 全面审视必须包含：评分表（多维度）+ 硬伤清单 + 趋势对比 + 建议优先级
- 针对性审视：聚焦报告，列出发现的问题和建议
- 快速审视：可在工作日志中简要记录

**4. 向用户展示结论并获确认**
- 展示审视结论（评分、硬伤、建议）等待用户审阅
- 用户确认后方可进入落笔同步阶段
- 用户可能要求调整重点或补充分析

**5. 落笔同步 —— 审视后输出物清单**

审视完成后，按审视级别同步对应的文档：

| 输出物 | 🔬 全面审视 | 🎯 针对性审视 | ⚡ 快速审视 |
|:-------|:----------:|:------------:|:----------:|
| AI 工作日志 `docs/ai-work-logs/` | ✅ 完整报告 | ✅ 记录发现 | ✅ 简要记录 |
| 债务日志 `技术债务与架构决策/技术债务日志.md` | ✅ 新债登记+状态变更 | ✅ 按需 | ❌ |
| 项目看板 `docs/计划/README.md` | ✅ 更新统计+待办 | 按需 | ❌ |
| 会话交接文档 `docs/流程规范/AI会话交接文档.md` | ✅ 更新§2/§3/§5 | 按需 | ❌ |
| 日志 README 索引 `docs/ai-work-logs/README.md` | ✅ 新增索引行 | ✅ 按需 | 按需 |
| ADR `技术债务与架构决策/架构决策记录.md` | 按需（架构变更时） | 按需 | ❌ |

**6. 代码审查**（仅审视涉及代码变更时）
- 如果审视导致了代码修改，走 §3.1 功能开发流程的代码审查步骤
- 审视本身通常不涉及代码变更（修债务时除外）

**7. 提交与推送** — 按 §5.1 推送管理执行

## 4. 每次审视必查清单

> 此清单是 §3.5 审视工作流**第 2 步「系统检查」**的执行工具。
> 审视时根据审视级别（全面/针对/快速）选择对应的层面逐一检查。
> 这是发现债务的主要途径。

### 4.1 架构层面

- [ ] 是否有模块缺少必要的抽象层？
- [ ] 模块间接口是否稳定？是否需要新的 ADR？
- [ ] 现有 ADR 是否仍然有效？是否有被新情况推翻的？
- [ ] 依赖项是否有版本冲突或过时的？

### 4.2 代码层面

- [ ] 是否有未处理的 `# TODO` / `# FIXME` / `# HACK` 注释？
- [ ] 是否有函数签名缺少类型注解？
- [ ] 是否有 `dict = None` 这类 Pyright 不通过的写法？
- [ ] 是否有硬编码的配置值（API Key、URL、价格）？
- [ ] 是否有重复代码段可以抽取？
- [ ] 是否有 import 未使用或排序混乱？

### 4.3 测试层面

- [ ] 是否有模块缺少对应的测试文件？
- [ ] 新的功能代码是否可测试？是否需要 Mock 工具？
- [ ] CI 流水线是否正常通过？

### 4.4 文档层面

- [ ] 新增功能是否有对应的文档说明？
- [ ] 配置项变化是否更新了 `.env.example`？
- [ ] 项目看板（`docs/计划/README.md`）是否需要更新？
- [ ] AI 工作日志是否需要更新？
- [ ] 债务日志是否需要更新？
- [ ] 交接文档（`docs/流程规范/AI会话交接文档.md`）是否需要同步更新？
- [ ] ADR（`docs/技术债务与架构决策/架构决策记录.md`）是否需要新增或更新？

### 4.5 安全层面

- [ ] 是否有敏感信息可能泄露（API Key、密码）？
- [ ] 是否有未鉴权的操作？
- [ ] 日志中是否会打印密钥等敏感信息？

## 6. 债务管理流程

### 6.1 发现即登记

审视中发现的任何债务，**立即登记**，三步完成：

```
① 复制模板（债务新增模板.md）→ ② 填写四维分类 → ③ 追加到债务日志
```

### 6.2 每天更新规则

| 时段 | 动作 | 说明 |
|------|------|------|
| 🟢 **无新债务** | 日志中备注"本次未发现新债务" | 保持习惯 |

### 3.6 双模式选择指南：轻量模式 vs 完整模式

> 核心原则：**checklist 的质量底线不因模式不同而打折，压缩的是仪式性开销，不是纪律。**

#### 模式对比

| 维度 | 🚀 轻量模式（日常开发） | 🏛️ 完整模式（架构变更） |
|:----|:----------------------|:----------------------|
| **适用场景** | 模块内增删改、接口接驳、bug 修复、文档更新 | 新增模块、跨模块重构、技术选型、架构调整 |
| **判断标准** | 不新增文件 AND 不修改跨模块接口 **AND 非关键代码路径** | 新增文件 OR 修改接口契约 OR 涉及多模块 **OR 关键代码路径** |
| **需求确认** | 口头确认（"是不是这样？"） | full brainstorming（边界条件逐条过） |
| **实现计划** | 会话内 3-5 行 bullet | `docs/superpowers/plans/` 完整文档 |
| **测试** | TDD 节奏不变，但不强制每步三段代码 | 完整 RED→GREEN→IMPROVE 循环 |
| **代码审查** | 手动扫一遍 + Ruff/Pyright | 完整子 Agent 审查或对照 §4 清单 |
| **全量测试** | ✅ **必须跑** | ✅ **必须跑** |
| **三同步** | ✅ **代码+测试+文档+债务，改多少同步多少** | ✅ **同上** |
| **上下文开销** | ~70% 时间在写代码 | ~50% 时间在设计+规划 |

#### 模式选择树

```
本次任务是什么？
│
├─ 模块内新增函数 / 修 bug / 改文档
│  → 🚀 轻量模式
│
├─ 模块间接驳 / 新增接口
│  ├─ 接口契约不变（只扩展） → 🚀 轻量模式
│  └─ 改接口签名 / 改 Pydantic 模型 → 🏛️ 完整模式
│
├─ 新增一个完整模块
│  → 🏛️ 完整模式
│
└─ 跨模块重构 / 技术选型切换
    → 🏛️ 完整模式（需要 ADR）
```

#### 关键代码判断准则

以下场景**必须走完整模式**，即使只是小改动：

| 场景 | 原因 |
|:----|:------|
| LLM 调用逻辑（`src/utils/llm.py`） | 直接影响模型输出质量，改错引发系统性幻觉 |
| 模块间数据契约（Pydantic `BaseModel`） | 一改影响上下游所有模块，遗漏一处就是 silent bug |
| Agent 编排主流程（`orchestrator.py`、LangGraph 图） | 编排出错 → 整个分析链路报废 |
| 资金/风控相关计算 | 数值错误造成实际决策误导 |
| 对外暴露的 API / CLI 接口 | 破坏向后兼容，影响所有调用方 |
| 模型负载或推理的 Memory/State 管理 | 上下文泄漏/状态错误难排查 |
| 缓存策略（TTL、失效逻辑） | 静默返回过期数据比不返回更危险 |

> **简记**：涉及钱、模型输出、数据流通、外部接口的，轻量模式自动升为完整模式。
```

#### 质量底线（两模式均不可违反）

```
┌────────────────────────────────────────────────────────┐
│  🔴 以下环节在任何模式下都不可跳过：                     │
│                                                        │
│  1. 新功能必须有对应测试                                 │
│  2. make check（lint + type + test）提交前必须全通过     │
│  3. 三同步原则：代码 + 测试 + 文档 + 债务                 │
│  4. 新增债务必须登记                                     │
│  5. 会话结束必须更新工作日志                              │
└────────────────────────────────────────────────────────┘
```

> 判断不清时走完整模式。宁可多花 15 分钟规划，不要花 2 小时返工。

## 8. 突发情况处理

### 8.1 遇到阻塞

```markdown
1. 记录阻塞原因到 AI 工作日志
2. 如果是代码问题 → 评估是否要新增债务
3. 如果是外部依赖（API 不可用等）→ 记录等待
4. 明确告知用户阻塞了什么，建议下一步
```

### 8.2 上下文耗尽

```markdown
1. 执行会话结束 checklist（§5.1）
2. 确保 AI 工作日志完整
3. 确保债务日志已更新
4. 创建 `docs/流程规范/会话交接提示.md` — 新会话可直接粘贴的启动提示
   - 包含：本次未完成的核心事项、关键项目事实避免重复查询、关键文档索引
   - 包含：后续开发路线图和红线提醒
5. 在日志末尾标记「上下文耗尽，续接」
```

### 8.3 发现之前的决策有误

```markdown
1. 更新相关 ADR（标记为 ⏪ 已替换）
2. 创建新的 ADR 记录新的决策
3. 在 AI 工作日志中说明变更理由
4. 更新受影响的代码和文档
```

### 8.4 认知整理与结构化归档

> 当会话产生了新的行业认知、竞品分析、架构对照等**非代码类成果**时，
> 应在会话结束前将其结构化归档到功能模块文件夹中。

**适用场景**：
- 调研了某个竞品项目 → 分析笔记存入对应模块的源码分析子文件夹
- 对比了某种架构方案 → 设计方案存入对应模块的架构设计子文件夹
- 发现了某个数据源的坑 → 踩坑记录存入对应模块的子文件夹
- 阅读了某篇论文 → 笔记存入对应模块的文件夹

**操作步骤**：

```markdown
1. 确定成果属于哪个功能模块（01-数据采集 / 02-辩论决策引擎 / ... / 09-人格与提示词）
2. 进入对应模块文件夹，看 README 确认子文件夹分类
3. 将成果写入对应子文件夹（如果没有合适的子文件夹就新建一个）
4. 如果成果跨越多个模块或在模块之外：
   - 写入 `docs/调研分析/` 下的主文档
   - 在 `docs/调研分析/行业开源方案对照知识库.md` 中补充战线分析
5. 更新功能模块 README 的"关键研究问题"（如有新增问题）
```

**为什么重要**：

```
不归档的认知 = 下次会话重新调研（浪费 token）
归档的结构化认知 = 下次直接使用（积累效应）

好的认知管理体系应该让每个会话都能站在上个会话的肩膀上。
```

**归档位置速查**：

| 成果类型 | 归档位置 |
|---------|---------|
| 竞品源码分析 | `功能模块/NN/XXX源码分析/` |
| 架构设计方案 | `功能模块/NN/架构设计/` |
| 踩坑记录 | `功能模块/NN/踩坑记录/` |
| 跨模块/行业格局 | `docs/调研分析/行业开源方案对照知识库.md` |
| 论文阅读笔记 | `功能模块/NN/对应子文件夹/` |
