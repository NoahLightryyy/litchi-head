# 📐 上下文加载策略（Context Routing）

> **用途**：定义 AI 会话中按任务类型选择加载哪些文档、按什么顺序加载、分配多少上下文预算。
>
> **解决的核心问题**：docs/ 约 70+ 文件。次次全量加载 → 上下文窗口在启动阶段就消耗 30%+ 容量。
> 按任务路由文档 × 渐进披露 → 启动阶段只占 ~8K tokens（memory + 当前状态）。
>
> **阅读时机**：
> - 🆕 新 AI 首次接入项目（配合 CLAUDE.md 理解加载体系）
> - 🔧 新增/变更了 docs/ 目录结构后更新本文档
> - 📋 需要确认某个特定任务的加载策略
> - **日常开发不需要重读本文档**（`/resume-session` Skill 已封装加载策略）
>
> **设计参照**：Claude Skills 渐进披露 / Cursor `.cursor/rules` 条件加载 / Cline Memory Bank 三层模型。

---

## 1. 三层加载体系

```
┌─────────────────────────────────────────────────────┐
│  L1 ─ 自动注入（Memory 层）          常驻 ~2K tokens │
│    不必手动加载，Memory 系统在每个会话自动注入         │
│    ├── project-identity      → 技术栈、红线、定位     │
│    ├── architecture-decisions → ADR-001~010 要点摘要  │
│    └── current-state          → 活跃任务、模块完成度   │
├─────────────────────────────────────────────────────┤
│  L2 ─ 快速同步（每次会话必读）    ~5~8K tokens        │
│    resume-session Skill 自动完成                     │
│    ├── HANDOVER.md §2（当前状态）+ §5（下一步）       │
│    ├── 最新 AI 工作日志                              │
│    ├── git status（工作区快照）                       │
│    └── pytest --collect-only -q（测试数验证）         │
├─────────────────────────────────────────────────────┤
│  L3 ─ 按需加载（按任务类型路由）    ~5~20K tokens     │
│    ↓ 见 §2 任务路由矩阵，根据当前任务选择加载          │
└─────────────────────────────────────────────────────┘
```

### 1.1 为什么不是全量加载

| 方案 | 启动 token | 问题 |
|:----|:----------:|:-----|
| ❌ 全量加载 9 个模块 SPEC | ~20K | 写了 ROUTING.md 但 AI 不看，等于白写 |
| ❌ 全量加载所有 ADR + 债务 | ~15K | 启动即占 35K，真正工作只剩几十 K |
| ✅ **按任务路由** | ~8K | 只加载当前任务需要的，剩下的用到再读 |

### 1.2 加载优先级等级

```
S-tier（必须）  ─ 不读无法工作          → HANDOVER.md §2+§5、最新日志
A-tier（重要）  ─ 读了少踩坑            → 对应模块 SPEC + ADR
B-tier（按需）  ─ 遇到问题才读          → 债务日志相关条目、模块 README
C-tier（跳过）  ─ 当前任务不需要        → 归档、非相关模块文档、旧日志（30天前）
```

## 2. 任务路由矩阵

> **使用方式**：根据当前任务类型，在对应行找到「必读」「按需」「跳过」列，逐项加载。
>
> 矩阵中提到的文件路径均为 `docs/` 下相对路径。

| # | 任务类型 | 🟢 必读（S+A 级） | 🟡 按需（B 级） | 🔴 跳过（C 级） |
|:-:|:---------|:------------------|:-----------------|:----------------|
| 1 | **继续开发**<br>（在现有模块上增删改） | `01-guides/HANDOVER.md §2+§5`<br>`04-changelog/logs/`（最新一篇）<br>对应模块 `03-modules/NN-xxx/SPEC.md` | 对应模块 `ADR.md`<br>对应模块 `README.md` | 非本模块文档<br>99-archive/<br>30 天前的日志 |
| 2 | **新增功能**<br>（新增文件/接口） | 对应模块 `03-modules/NN-xxx/SPEC.md`<br>对应模块 `ADR.md` | 相关模块接口模型<br>`05-decisions/`（跨模块 ADR） | 工作日志<br>债务日志<br>非相关模块 |
| 3 | **Bug 修复** | 对应模块 `SPEC.md`<br>对应模块测试文件 | 债务日志相关条目<br>（`01-guides/debt/*.md`） | 架构设计<br>ADR（除非涉及架构）<br>非本模块文档 |
| 4 | **架构设计** | `05-decisions/README.md`（ADR 索引）<br>涉及模块的 `ADR.md` | 跨模块接口契约<br>（`src/core/protocol.py` 等） | SPEC 实现细节<br>工作日志 |
| 5 | **代码审查** | `01-guides/WORKFLOW.md §4`（审视清单）<br>受审模块 `SPEC.md` | 受审模块 `ADR.md` | 所有非本模块文档<br>`99-archive/` |
| 6 | **审视/审计** | `01-guides/WORKFLOW.md §4`<br>全部 `01-guides/debt/*.md` | 各模块 `SPEC.md`（仅 README 级摘要）<br>`05-decisions/` | `99-archive/` |
| 7 | **文档更新** | `README.md`（docs 路由表）<br>被更新文档的上下游引用 | 相关模块 `SPEC.md` | 功能代码<br>测试代码<br>债务日志 |
| 8 | **调研分析** | `99-archive/` 对应条目<br>`README.md` 路由表 | 竞品源码分析笔记<br>对应模块 SPEC 的战线分析 | 债务日志<br>工作日志 |
| 9 | **部署/CI** | `01-guides/ENVIRONMENT.md`<br>`.env.example` | CI 配置 `.github/workflows/` | 不加载文档，直接看 Makefile |
| 10 | **会话恢复**<br>（`/resume-session`） | L1 Memory + L2 快速同步<br>参见 §3.1 完整流程 | 按下一步任务切到对应行 | 所有离线的 |

## 3. 上下文预算分配

> 每个任务的上下文预算不是硬限制，而是**指导原则**——超过了可以加，
> 但超出意味着需要确认是否"真的需要这么多文档"。

### 3.1 预算速查

| 任务规模 | 文档预算 | 模型建议 | 典型场景 |
|:--------:|:--------:|:---------|:---------|
| 🟢 **简单** | ~8K | `deepseek-chat` | 单文件编辑、文档更新、重启开发 |
| 🟡 **中等** | ~15K | `deepseek-chat` | 模块内新增功能、Bug 修复、代码审查 |
| 🟠 **复杂** | ~25K | `deepseek-chat`（可切 v4-pro） | 跨模块改动、架构设计、审视审计 |
| 🔴 **极复杂** | ~40K+ | `deepseek-v4-pro` | 新增模块、系统重构、安全审查 |

> **模型路由联动**：遇到复杂任务（文档预算 > 20K），考虑切 `deepseek-v4-pro`。
> 详细策略见 [CLAUDE.md 模型策略](../../CLAUDE.md#模型策略快慢分离)。

### 3.2 上下文预算使用原则

```
1. 先读 S 级（必须），再用剩余预算读 A 级（重要）
2. B 级文档只在具体问题出现时才读（不要预读）
3. 如果文档预算超过 20K，考虑用 subagent 做隔离任务
4. 每次读文档前问自己：这个任务真的需要这份文档吗？
5. 同一次会话中切换任务类型 → 重新评估预算分配
```

## 4. 文档分类与衰减

> 不同文档的"保质期"不同。旧内容占用上下文 = 浪费。

| 类别 | 示例 | 有效周期 | 加载策略 |
|:-----|:-----|:--------:|:---------|
| 🔒 **永久** | 架构红线、技术栈约定 | 永久 | 全量读一次，除非架构变更 |
| 📅 **会话** | 当前状态、当前模块 SPEC | 本次会话 | 每次会话开头加载 |
| 🔄 **周期性** | AI 工作日志、测试统计 | 最近一次 | 只读最新，旧日志不加载 |
| ⏳ **一次性** | 债务日志条目、调研笔记 | 用后即弃 | 需要时才加载，用完即释放 |
| 🗑️ **过期** | 30 天前的日志、已归档设计 | 不加载 | 99-archive/ 仅调研时访问 |

### 4.1 日志衰减规则

```
最新日志         → 必读（S 级）
前一次日志       → 仅当提到"上下文耗尽"时读
30 天前的日志    → 不看（除非需要查找历史决策）
```

## 5. 规则冲突与例外

### 5.1 加载冲突处理

当多个任务类型叠加（如"修 Bug 的同时涉及架构变更"）：

```
1. 以"更重"的任务类型为准      ← 修 Bug 涉及架构 → 走架构设计的加载策略
2. 任务不明确时走"审视/审计"    ← 安全策略：多读比少读好
3. 上下文不足时降级            ← 按 S > A > B 优先级砍
```

### 5.2 例外：安全相关任务

任何涉及以下内容的任务，**自动升一级加载策略**：

```
- 认证/授权代码
- API Key 管理
- 资金计算逻辑
- LLM 调用封装（src/utils/llm.py）
- 模型输出解析
```

升一级 = 简单→中等→复杂→极复杂，同时追加 `01-guides/WORKFLOW.md §4.5` 安全清单。

### 5.3 例外：上下文耗尽

```
检测到上下文接近耗尽（~20K tokens 剩余）→ 立即执行 CLAUDE.md 交接流程：
  1. 停止加载任何新文档
  2. 只写日志、更新债务、提交
  3. 标记 context-exhausted-handover
```
