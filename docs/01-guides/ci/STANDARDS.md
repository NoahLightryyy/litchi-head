# CI 标准规范

> **读给**：全部开发者
> **归属**：[🔄 CI 治理部](../../06-departments/11-ci-governance/ROLE.md)
> **原则**：CI 永远是绿的。红就是事故，立即处理。

---

## 1. CI 检查项定义

当前 `.github/workflows/ci.yml` 包含以下检查：

| # | 检查项 | 工具 | 门禁级别 | 最大耗时 |
|:-:|:-------|:----|:--------:|:--------:|
| 1 | 代码风格 | Ruff | 🔴 阻塞 | ~10s |
| 2 | 类型检查 | Pyright | 🔴 阻塞 | ~30s |
| 3 | 测试 | Pytest | 🔴 阻塞 | ~60s |
| 4 | 覆盖率 | pytest-cov (≥80%) | 🔴 阻塞 | ~60s |
| 5 | 依赖漏洞 | pip-audit | 🟡 警告 | ~30s |

### 门禁级别

| 级别 | 含义 | 处理 |
|:----:|:-----|:-----|
| 🔴 阻塞 | 合并前必须修复 | 不可合并、不可推 main |
| 🟡 警告 | 不阻塞构建，但应关注 | 记录到 CI 债务，下次迭代修 |
| 🟢 信息 | 仅供参考 | 可忽略 |

---

## 2. CI 状态定义

### 🟢 绿 — 全部通过

所有 🔴 阻塞检查通过。可安全合并。

### 🟡 黄 — 有警告

🔴 检查全过，但 🟡 警告项有失败。可合并，但应在下次迭代处理。

### 🔴 红 — 有失败

任一 🔴 阻塞检查失败：
- **阻止合并到 main**
- **阻止推送新功能**
- **处理优先级高于一切新功能开发**

## 3. CI 纪律

### 推送前

```bash
# 必须运行（本地能过的不要让 CI 抓）
make check      # = ruff check . + pyright src/ + pytest -v --tb=short
```

### 推送后

- 等待 GitHub Actions 完成（约 2-3 分钟）
- 如变红 → 立即执行 [CI 处理工作流](WORKFLOW.md)
- **不允许在 CI 红着时推送下一个提交**

### Batch Loop 中

Batch Loop 收尾前，自动执行 CI 健康自检（`ruff check .` + `pyright src/`），参见 [CLAUDE.md](../../../CLAUDE.md#batch-loop-模式)：

```
├── CI 健康自检（ruff + pyright，有红先修）     ← 全自动
```

### 异常情况

| 场景 | 处理 |
|:-----|:-----|
| CI 红了但急需提交 | 建分支提交，修好 CI 后再合并 |
| CI 检查太慢 | 联系 CI 治理部优化 |
| 新增检查项 | CI 治理部经讨论后修改 workflow |

## 4. 多 Python 版本策略

CI 对 Python 3.12 / 3.13 双版本运行：

| 版本 | 策略 |
|:----:|:-----|
| 3.12 | 主版本 —— 全部检查通过才算绿 |
| 3.13 | 超前兼容 —— 允许 Pyright 超时/失败不阻塞（但应跟踪） |

> 3.13 的 Pyright 超时已知问题，见 [TROUBLESHOOTING.md](TROUBLESHOOTING.md#pyright-在-python-313-上超时)
