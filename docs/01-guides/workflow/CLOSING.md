## 0. 何时执行 — 三种触发场景

> **核心纪律**：以下三种场景**都**要执行收尾。不是只有"会话结束"才跑。

| 场景 | 执行级别 | 触发条件 |
|:-----|:---------|:---------|
| 🏁 **功能交付**（最常见） | **强制闸门**（§1） | 一个功能的最后一个 Task 提交后，问用户"下一步"之前 |
| 🛑 **上下文耗尽** | **完整收尾**（§2） | 上下文剩余 ~20K tokens，必须交接 |
| ✅ **会话结束** | **完整收尾**（§2） | 用户说"今天到这"或主动停止 |

**纪律红线**：功能交付后、问用户"下一步"之前，**必须先过§1强制闸门**。
不过闸门就问"下一步"= 流程违规，用户发现应纠正。

## 1. 强制闸门（每次功能交付后执行）

> 一个功能的最后一个 Task 提交后、问用户"下一步"之前，**必须过此闸门**。
> 耗时约 2-3 分钟。

### 执行步骤

```
┌─ 1. CI 自检 ────────────────────────────────────┐
│  ruff check .（代码风格）                          │
│  pyright src/   （类型检查）                        │
│  小问题顺手修，修不了的看一眼是否值得记债务            │
├─ 2. 改动模块测试 ────────────────────────────────┤
│  pytest <本次改动文件的对应测试文件> -v              │
│  ⚠️ 只跑改动的模块，全量测试只有在以下情况才需要：     │
│     - 改动了跨模块共享基础设施（llm.py/config.py）     │
│     - 改动了 pytest fixture/conftest                │
│     - 用户明确要求                                   │
├─ 3. 发文更新 ────────────────────────────────────┤
│  ✓ 学习卡片（如有新技术/设计模式）                    │
│  ✓ AI 工作日志（docs/04-changelog/logs/）            │
├─ 4. 问用户 ────────────────────────────────────┤
│   "XX 完成了。下个方向？"                            │
└────────────────────────────────────────────────────┘
```

**常见违规**（已发生过，记录在此）：
| 违规 | 后果 | 纠正 |
|:-----|:-----|:------|
| Task 3 提交后直接问"下个做啥" | 日志/看板/CI 未同步 → 用户催收尾 | 强制闸门收口 |
| 自动跑全量测试 | 浪费 3min+，用户不满 | 只跑改动模块，全量需批准 |

### 失败处理

| 闸门检查失败 | 处理方式 |
|:------------|:---------|
| Ruff error | 自动 `ruff check --fix` 修，修不好→登记新债务 |
| Pyright error | 修，修不好→登记新债务后标记 ⚠️ 已知类型问题 |
| 测试失败 | 必须修，修好再过闸 |
| CI 状态异常 | 标记到输出，建议优先处理 |

## 2. 完整收尾 checklist（会话结束/上下文耗尽时执行）

> **上下文耗尽时**：检测到上下文接近上限（~20K tokens 剩余）时立即执行本 checklist，
> 不推进新工作。详细流程见 [EMERGENCIES.md](EMERGENCIES.md) §2.2。

```markdown
## 会话结束 checklist

### ☑️ 文档更新
- [ ] AI 工作日志已更新（`docs/04-changelog/logs/YYYY-MM-DD/YYYY-MM-DD.md`）
- [ ] 日志 README 索引已更新（如有新增日志文件）
- [ ] **已执行文档同步审计清单**（见 DEVELOPMENT.md §2，按触发条件逐项核查）
- [ ] **本部门 HANDOVER.md 已更新**（`docs/06-departments/{id}/HANDOVER.md`）
- [ ] **本部门 DEBT.md 已更新**（如有债务变更）
- [ ] 跨部门债务 → `docs/06-departments/00-cross-cutting/DEBT.md` 同步
- [ ] 债务路由 `docs/01-guides/debt/ROUTER.md` 计数同步
- [ ] 相关的设计文档已同步更新
- [ ] ADR 已更新（如有架构变更）
- [ ] **引用清理**：如本次创建了新文件/新方案，已搜索旧方案的残留引用并清理

### ☑️ 上下文耗尽专属（自动触发）
- [ ] 日志末尾标记「上下文耗尽，需续接」
- [ ] 所有未完成的 todo 已记录到日志「待办事项」
- [ ] 当前 Batch 进度已记录（做到哪一步、卡在哪、下一步计划）
- [ ] Git 已提交，commit 信息含 `context-exhausted-handover` 标记
- [ ] 新会话启动提示已输出

### ☑️ 审视专属检查（仅审视类会话）
- [ ] 审视结论已向用户完整展示
- [ ] 用户已确认审视结论
- [ ] 审视输出物清单已按 DEVELOPMENT.md §9 级别要求全部完成

### ☑️ 代码质量
- [ ] Ruff 检查通过
- [ ] Pyright 检查通过
- [ ] Pytest 所有测试通过
- [ ] Git 工作区干净或有合理的未提交变更

### ☑️ 推送管理
- [ ] 已选择推送策略：`git push` 已执行 / 标记「待推送」
- [ ] 如已推送，确认远程 CI 通过（GitHub Actions）→ [CI 治理](../ci/README.md)
- [ ] 如待推送，记录 ahead commits 数量

### ☑️ 任务闭环
- [ ] 已完成本次目标任务
- [ ] 未完成的部分记录在日志的「待办事项」中
- [ ] 阻塞项已标记清楚（🔴）
```

## 4. 债务管理流程

> **债务已按部门拆分**。每个部门的债务在各 `docs/06-departments/{id}/DEBT.md` 中管理。
> 跨部门的债务在 `docs/06-departments/00-cross-cutting/DEBT.md` 中管理。
> 全局路由 `docs/01-guides/debt/ROUTER.md` 只保留统计仪表盘。

### 4.1 发现即登记 — 三步完成

```
① 判断归属 → 改哪个 `src/{dir}/` 的代码产生的？
   │
   ├─ 只影响一个部门 → 加在该部门的 DEBT.md
   ├─ 影响多个部门 → 加在 00-cross-cutting/DEBT.md
   └─ 不太确定 → 加在 00-cross-cutting/DEBT.md（后续有人会调整）
   │
   ▼
② 按模板填写（ID / 标题 / 严重度 / 状态 / 描述）
   │
   ▼
③ 更新部门 DEBT.md + 全局 ROUTER.md 计数同步
```

### 4.2 每天更新规则

| 时段 | 动作 | 说明 |
|------|------|------|
| 🟢 **无新债务** | 日志中备注"本次未发现新债务" | 保持习惯 |
| 🟡 **有新债务** | 追加到该部门的 DEBT.md + 更新 ROUTER.md 计数 | 按模板填写 |
| 🟠 **状态变更** | 更新 DEBT.md 中的状态 + 关联 HANDOVER.md | 与工作日志联动 |
| 🔴 **关闭债务** | 标记 ✅ + 关闭日期 + 修复说明 | 更新 ROUTER.md 仪表盘 |

### 4.3 利息定期复审

| 状态 | 复审频率 | 动作 |
|------|---------|------|
| 📖 已接受 | 每月 | 检查利息是否超标，需要重新排期？ |
| 🆕 待评估 | 每周 | 积压太多要批量评估 |
| 📋 已确认 | 每 Sprint | 确认排期优先级 |

## 5. 日志更新规范

### 5.1 AI 工作日志

- **文件名**：`docs/04-changelog/logs/YYYY-MM-DD/YYYY-MM-DD.md`
- **单日多次对话**：`YYYY-MM-DD-N.md`（N=1,2,3...）
- **每次会话都写**：即使只做了小改动
- **内容要求**：记录做了什么、改了什么、为什么、涉及部门、下一步建议

### 5.2 债务日志

- **按日期分组**：部门 DEBT.md 按发现日分组展示
- **按状态管理**：按生命周期状态分组（待评估/已确认/修复中等）
- **新债追加**：对应分组 + 对应日期
- **状态迁移**：债务从"已确认"→"修复中"时，更新分组位置

### 5.3 日志间交叉引用

```markdown
AI 工作日志  ←引用→  债务日志（记录新增/变更了哪些债务）
债务日志     ←引用→  AI 工作日志（"修复方向参见某日日志"）
```

## 6. 项目看板更新规范

**看板位置**：`docs/00-overview/ROADMAP.md`

### 更新时机

- 审视会话后：必须更新「快速统计」和「待办」状态
- 功能开发完成：更新模块完成度和测试统计
- 债务关闭后：更新紧急指数和开放债务数

### 更新内容

- 总测试数、技术债务数、紧急指数、当前阶段
- 🟢 已完成 → 🔵 进行中 → ⬜ 待办的三段状态同步
- 模块完成度气温图（如新增了实现）

## 7. 部门交接文档更新规范

**文档位置**：

- 全局仪表盘：`docs/01-guides/HANDOVER.md`
- 部门状态：`docs/06-departments/{id}/HANDOVER.md`
- 部门债务：`docs/06-departments/{id}/DEBT.md`
- 跨部门状态/债务：`docs/06-departments/00-cross-cutting/HANDOVER.md` + `DEBT.md`

### 更新时机

- **每次会话结束**：更新涉及部门的 HANDOVER.md + DEBT.md
- 审视会话后：同步更新部门 HANDOVER.md 的"下一步优先级"
- 全局状态变更（测试数/CI）：更新 `00-cross-cutting/HANDOVER.md`

### 更新内容（部门 HANDOVER.md）

- 模块完成度 / 测试数
- 开放债务列表（与 DEBT.md 保持一致）
- 下一步优先级（清理债务的顺序）
- 页尾「最后更新」日期

### 同步规则

- 改了部门 DEBT.md → 确认 ROUTER.md 的计数同步
- 改了部门 HANDOVER.md → 确认全局 HANDOVER.md 的部门状态表同步
- 改了下游接口 → 确认 ROLE.md 的"对外接口"节是对的
