## 1. 功能开发标准步骤

需求确认 → 债务扫描 → 接口设计 → 实现 → 自测 → 文档 → 债务登记 → 代码审查 → 提交
   📋       🔍         📐       🔨    🧪    📝      🐛        🧐        📤

### 1.1 需求确认

理解要做什么，明确验收标准。

**⚠️ 战略意图落笔原则**：当用户表达了**产品方向/团队构建展望/架构哲学转向**等战略级意图时：
- 在写代码**之前**，必须先在 `docs/99-archive/` 中创建/更新设计文档，记录完整意图
- 不能只靠工作日志和交接文档的任务清单——任务清单是推演结果，不是原始意图
- 触发信号：用户说"我认为应该……""我的想法是……""这个项目的定位是……""展望""方向""哲学"等
- 反例（已修复）：2026-06-13 "从辩论团伙到专业交易团队"的战略转向只写进了工作日志和交接文档，缺少独立设计文档 → 补录为 `docs/99-archive/PROFESSIONAL-TEAM-OUTLOOK.md`

### 1.2 ⚡ 债务扫描

动手前先扫一眼本次改动范围的相关开放债务：
- 15 秒快速定位：看 `docs/01-guides/debt/ROUTER.md` 仪表盘
- 有直接相关的开放债务 → 在接口设计中考虑一并修复，避免方案选错
- 不在本次 scope 的 → 在日志中备注一句已知关联债务，避免重复工作
- **为什么要在接口设计之前做**：如果有个债务等着让接口做得不一样（如资金流向应走 Provider 层而非直接 akshare），接口设计错了后面全白做

### 1.3 接口设计

先定义 Pydantic 模型和函数签名（契约优先）。

### 1.4 实现

编写功能代码。

### 1.5 自测

至少跑一次手动验证，有测试则跑测试。

**测试范围策略**：

| 阶段 | 测试范围 | 命令 |
|:----|:---------|:-----|
| 🏗️ **日常开发** | 只跑本次改动**相关测试文件** | `pytest -v <文件路径>` |
| 🧹 **最终整合/大清理** | 全量测试 | `make test`（`pytest -v --tb=short`） |

> 日常逐次改动只跑全量会浪费大量时间（CI 已经会全量跑）。相关测试文件判断：
> - 改了 `src/data/providers/` → 跑 `tests/test_data/test_data_providers.py`
> - 改了 `src/debate/` → 跑 `tests/test_debate/`
> - 改了多个模块 → 跑所有受影响目录的测试文件

### 1.6 文档

更新相关 `.md` 文档，**必须执行下文的文档同步审计清单**。

### 1.7 债务登记

本次实现是否引入了新的债务？→ 登记；是否顺手修了已有债务？→ 关闭。
详见 [CLOSING.md](CLOSING.md) 的债务管理流程。

### 1.8 代码审查

使用 `code-reviewer` 代理审查变更；若涉及安全敏感代码先使用 `security-reviewer` 代理。

### 1.9 提交

规范的 git commit message。参见 `git-workflow.md`。

## 3. 战略决策捕获与检索

> **问题**：用户在 06-13 表达的「从辩论团伙到专业交易团队」战略转向，
> 只写进了工作日志和交接文档，没有形成独立设计文档。
> 后来用户问"我的构建展望写进文档了没"，需要全文搜索 30+ 文件才能确认。

### 3.1 决策分层

```
用户表达的意图
      │
      ├─ 战略级 (产品方向/团队构建/哲学转向)
      │   → docs/99-archive/ 独立文档
      │   → docs/99-archive/STRATEGIC-DECISIONS.md 登记条目
      │
      ├─ 架构级 (技术选型/模块设计)
      │   → ADR (docs/05-decisions/README.md)
      │
      ├─ 战术级 (具体实现方案)
      │   → 功能模块 README 的深挖方向/关键研究问题
      │
      └─ 执行级 (任务/债务)
          → 工作日志 + 债务日志 + 看板
```

### 3.2 三条铁律

1. **重要决策不散落** — 不能只写在工作日志里（工作日志只记"做了什么"，战略决策需要独立文档）
2. **搜索入口唯一** — 战略决策索引 `docs/99-archive/STRATEGIC-DECISIONS.md` 是"我们的方向定过什么"的唯一查询起点
3. **用户意图优先于 AI 推演** — 用户说的方向必须先落笔再实施，不能实施了就以为方向已记录

## 4. 代码质量门槛

**推送前 → hook 自动 `ruff + pyright + 快测试子集`**（~70s），**23 个 `@pytest.mark.slow` 慢测试由 CI 执行**。日常开发中按以下节奏：

| 时机 | 检查项 | 命令 |
|:----|:-------|:-----|
| 🔨 **每次修改后** | Ruff 代码风格 + Pyright 类型检查 | `make lint && make type` |
| 🔨 **每次修改后** | 跑本次改动**相关测试** | `pytest <相关文件路径>` |
| 📤 **推送前（hook 自动）** | 代码风格 + 类型 + 快测试子集（~70s） | `ruff` + `pyright` + `pytest -m "not slow" -x` |
| 🔬 **CI（GitHub Actions）** | 全量 lint + type + test（含慢测试） | 自动执行 |
| 📦 **大重构 / 架构变更** | 全量测试 + 覆盖率（可选） | `make check` |

**日常只跑相关测试，推送前 hook 自动拦截风格/类型/快测试失败，慢测试由 CI 负责。**

## 5. Agent 编排指引

根据任务类型，按需启用子代理配合执行：

| 场景 | 推荐代理 | 用途 |
|------|---------|------|
| 🏗️ 复杂功能/重构 | `planner` | 制定实现计划、分解任务 |
| 🔧 架构决策 | `architect` | 系统设计、方案对比 |
| 🧪 新功能 / Bug 修复 | `tdd-guide` | 先写测试（RED）→ 实现（GREEN）→ 重构（IMPROVE） |
| 🧐 代码刚写完 | `code-reviewer` | 代码质量审查（CRITICAL/HIGH 必须修复） |
| 🔒 安全敏感代码 | `security-reviewer` | 检查 OWASP Top 10、密钥泄露、注入风险 |
| 🔬 全面项目审视 | `architect` + `code-reviewer` | 多维度系统性审查 |
| 🐛 构建失败 | `build-error-resolver` | 分析错误日志、增量修复 |
| 📖 文档更新 | `doc-updater` | 更新 README / ADR / 工作日志 |

### Agent 不可用时降级方案

| 场景 | 默认 Agent | 降级方案 |
|------|-----------|---------|
| 🔍 代码审查 | `code-reviewer` | 手动审查，收紧标准（CRITICAL/HIGH 必修复） |
| 🏗️ 规划 | `planner` | 改用 EnterPlanMode 模式，手动分解任务 |
| 🧪 TDD | `tdd-guide` | 按照本文档 §1 标准步骤人工执行 |
| 🔒 安全审查 | `security-reviewer` | 运行 `bandit -r src/` 后手动审查结果 |
| 📖 文档 | `doc-updater` | 按模板手工创建/更新文件 |

> 降级不意味着降低质量门槛——审查标准不变，只是从 AI 助审变为人工审核。

## 6. 双模式选择指南

> 核心原则：**checklist 的质量底线不因模式不同而打折，压缩的是仪式性开销，不是纪律。**

### 6.1 模式对比

| 维度 | 🚀 轻量模式（日常开发） | 🏛️ 完整模式（架构变更） |
|:----|:----------------------|:----------------------|
| **适用场景** | 模块内增删改、bug 修复、文档更新 | 新增模块、跨模块重构、技术选型 |
| **判断标准** | 不新增文件 AND 不修改跨模块接口 | 新增文件 OR 修改接口契约 OR 跨模块 |
| **需求确认** | 口头确认 | full brainstorming |
| **实现计划** | 会话内 3-5 行 bullet | `docs/99-archive/` 完整文档 |
| **测试** | TDD 节奏不变；跑本次改动**相关测试** | 完整 RED→GREEN→IMPROVE；**全量测试** |
| **代码审查** | 手动扫一遍 + Ruff/Pyright | 完整子 Agent 审查或对照审视清单 |
| **测试范围** | 🏗️ 跑本次改动**相关测试文件** | 🧹 **全量 pytest**（`pytest -v --tb=short`） |
| **四同步** | ✅ **改多少同步多少** | ✅ **同上** |

### 6.2 模式选择树

```
本次任务是什么？
│
├─ 模块内新增函数 / 修 bug / 改文档
│  → 🚀 轻量模式
│
├─ 模块间接驳 / 新增接口
│  ├─ 接口契约不变（只扩展） → 🚀 轻量模式
│  └─ 改接口签名 / 改 Pydantic 模型 → 🏛️ 完整模式
│
├─ 新增一个完整模块
│  → 🏛️ 完整模式
│
└─ 跨模块重构 / 技术选型切换
    → 🏛️ 完整模式（需要 ADR）
```

### 6.3 关键代码判断准则

以下场景**必须走完整模式**，即使只是小改动：

| 场景 | 原因 |
|:----|:------|
| LLM 调用逻辑（`src/utils/llm.py`） | 直接影响模型输出质量，改错引发系统性幻觉 |
| 模块间数据契约（Pydantic `BaseModel`） | 一改影响上下游所有模块 |
| Agent 编排主流程（`orchestrator.py`、LangGraph 图） | 编排出错 → 整个分析链路报废 |
| 资金/风控相关计算 | 数值错误造成实际决策误导 |
| 对外暴露的 API / CLI 接口 | 破坏向后兼容 |
| 模型负载或推理的 Memory/State 管理 | 上下文泄漏/状态错误难排查 |
| 缓存策略（TTL、失效逻辑） | 静默返回过期数据比不返回更危险 |

### 6.4 质量底线（两模式均不可违反）

```
🔴 以下环节在任何模式下都不可跳过：
1. 新功能必须有对应测试
2. push 前 hook 通过（`ruff + pyright`）+ 相关测试通过；全量 `make check` 由 CI 负责
3. 四同步原则：代码 + 测试 + 文档 + 债务
4. 新增债务必须登记
5. 会话结束必须更新工作日志
```

> 判断不清时走完整模式。宁可多花 15 分钟规划，不要花 2 小时返工。

## 7. DeepSeek 模型策略

### 7.1 快慢分离

| 模型 | 默认/按需 | 思考模式 | 适用场景 |
|------|:---:|:---:|------|
| `deepseek-chat` | **默认** | ❌ 无 | 日常开发（代码编辑、Git、文件读写、文档更新） |
| `deepseek-v4-pro` | 按需切换 | ✅ 推理 | 架构设计、根因分析、复杂重构、安全审查 |

```bash
# Windows 用户环境变量（默认值，日常使用）
ANTHROPIC_MODEL=deepseek-chat

# 复杂任务：改为 deepseek-v4-pro，重启 Claude Code
# 任务完成后：改回 deepseek-chat，重启 Claude Code
```

### 7.2 切换时机

```
本次任务是什么？
│
├─ 单文件编辑 / Git 操作 / 读文件 / 文档更新
│  → deepseek-chat（默认，不改环境变量）
│
├─ 多文件重构 / 新增模块 / 接口契约变更
│  → 视规模而定，中等以下用 deepseek-chat
│
└─ 架构设计 / 根因分析 / 安全审查 / 跨模块影响评估
   → 切 deepseek-v4-pro，完成后再切回
```

### 7.3 复杂度感知路由（代码中自动切换）

项目代码中通过 `src/utils/complexity_router.py` 自动判断任务复杂度：

```python
from src.utils.llm import llm_service

# 自动检测复杂度 + 路由（默认行为）
reply = await llm_service.ainvoke_auto("分析市场情绪", agent_name="analyst")
```

```python
from src.utils.llm import LLMConfig

# 显式强制推理模式
config = LLMConfig(model="deepseek-reasoner", reasoning_effort="high")
reply = await llm_service.ainvoke("复杂分析", llm_config=config)
```

## 8. 集成测试规范

| 规则 | 说明 |
|------|------|
| `@pytest.mark.integration` | 所有调用真实 API 的测试必须标记此 marker |
| `@skip_no_key` 装饰器 | 环境变量未配置时自动跳过，保证 CI 通过 |
| `tmp_path` 隔离 | 测试使用临时目录，不写真实数据目录 |
| 独立文件 | 集成测试与单元测试分文件存放（`test_integration_*.py`） |
| 无 `src/` 改动 | 新增集成测试原则上不应修改业务代码 |

```python
def _has_api_key() -> bool:
    try:
        from src.utils.config import settings
        return bool(settings.deepseek_api_key)
    except Exception:
        return False

skip_no_key = pytest.mark.skipif(
    not _has_api_key(),
    reason="需要 API Key（在 .env 中配置）",
)
```

## 9. 审视/审核工作流

> 审视是一个独立的工作类型，与功能开发并列。不是所有审视都需要相同的工作量，但所有审视都必须完成其级别对应的核心步骤。

### 9.1 审视级别

| 级别 | 适用场景 | 预估耗时 | 输出要求 |
|:----:|:---------|:--------:|:---------|
| 🔬 **全面审视** | 项目里程碑、阶段切换、长期未审视 | 30-60min | 完整审视报告 + 全部文档同步 |
| 🎯 **针对性审视** | 聚焦特定模块/债务/风险 | 10-20min | 聚焦报告 + 相关文档同步 |
| ⚡ **快速审视** | 日常状态检查、微小变更审查 | 5-10min | 仅更新工作日志 |

### 9.2 审视标准步骤

```
审视启动 → 系统检查 → 输出结论 → 用户确认 → 落笔同步 → 代码审查 → 提交与推送
   📋       🔍        📝        ✅        ✍️        🧐        📤
```

**1. 审视启动** — 明确审视范围和级别
**2. 系统检查** — 按审视级别执行对应的 [审视清单](EMERGENCIES.md#1-审视必查清单)
**3. 输出审视结论** — 全面审视必须包含评分表 + 硬伤清单 + 趋势对比
**4. 向用户展示结论并获确认**
**5. 落笔同步** — 审视完成后的输出物清单：

| 输出物 | 🔬 全面 | 🎯 针对 | ⚡ 快速 |
|:-------|:-------:|:-------:|:-------:|
| AI 工作日志 | ✅ 完整报告 | ✅ 记录发现 | ✅ 简要记录 |
| 债务日志 | ✅ 新债登记+状态变更 | ✅ 按需 | ❌ |
| 主 README | ✅ 同步 badge/架构图 | ✅ 按需 | ❌ |
| 项目看板 | ✅ 更新统计+待办 | 按需 | ❌ |
| 会话交接文档 | ✅ 更新 | 按需 | ❌ |
| 日志 README 索引 | ✅ 新增索引行 | ✅ 按需 | 按需 |
| ADR | 按需 | ❌ | ❌ |

**6. 代码审查**（仅审视涉及代码变更时）
**7. 提交与推送** — 按 [CLOSING.md](CLOSING.md) 推送管理执行

## 10. 认知整理与结构化归档

> 当会话产生了新的行业认知、竞品分析、架构对照等**非代码类成果**时，应在会话结束前将其结构化归档。

### 适用场景

- 调研了某个竞品项目 → 分析笔记存入对应模块文件夹
- 对比了某种架构方案 → 设计方案存入对应模块文件夹
- 发现了某个数据源的坑 → 踩坑记录存入对应模块文件夹
- 阅读了某篇论文 → 笔记存入对应模块文件夹

### 操作步骤
