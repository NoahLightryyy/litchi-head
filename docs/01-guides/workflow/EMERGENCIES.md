## 1. 审视必查清单

> 此清单是 [DEVELOPMENT.md](DEVELOPMENT.md) §9 审视工作流**第 2 步「系统检查」**的执行工具。
> 审视时根据审视级别（全面/针对/快速）选择对应的层面逐一检查。
> 这是发现债务的主要途径。

### 1.1 架构层面

- [ ] 是否有模块缺少必要的抽象层？
- [ ] 模块间接口是否稳定？是否需要新的 ADR？
- [ ] 现有 ADR 是否仍然有效？是否有被新情况推翻的？
- [ ] 依赖项是否有版本冲突或过时的？

### 1.2 代码层面

- [ ] 是否有未处理的 `# TODO` / `# FIXME` / `# HACK` 注释？
- [ ] 是否有函数签名缺少类型注解？
- [ ] 是否有 `dict = None` 这类 Pyright 不通过的写法？
- [ ] 是否有硬编码的配置值（API Key、URL、价格）？
- [ ] 是否有重复代码段可以抽取？
- [ ] 是否有 import 未使用或排序混乱？
- [ ] **每个 `except` 块是否有日志记录？是否有静默 `pass`？** ← 实盘红线
- [ ] **每个外部调用（数据源/LLM）是否有超时/重试/fallback？** ← 实盘红线
- [ ] **函数在边界条件（空输入、None、网络断）下的行为是否定义且可观测？** ← 实盘红线
- [ ] **是否存在"看起来正常但数据是空的"的静默降级路径？** ← 实盘红线

### 1.3 测试层面

- [ ] 是否有模块缺少对应的测试文件？
- [ ] 新的功能代码是否可测试？是否需要 Mock 工具？
- [ ] CI 流水线是否正常通过？
- [ ] **每个函数是否有空输入/异常输入的测试用例？** ← 实盘红线
- [ ] **数据源断连、LLM 超时等外部依赖失败是否有模拟测试？** ← 实盘红线
- [ ] **后端 API 是否每个端点都有集成测试？** ← 实盘红线

### 1.4 文档层面

> 修改代码后的文档同步，优先执行 [DEVELOPMENT.md](DEVELOPMENT.md) §2 文档同步审计清单。

- [ ] 新增功能是否有对应的文档说明？
- [ ] 配置项变化是否更新了 `.env.example`？
- [ ] **主 README（`README.md`）** — 测试 badge/架构图/模块表/Phase 进度是否过期？
- [ ] 项目看板（`docs/00-overview/ROADMAP.md`）是否需要更新？
- [ ] AI 工作日志是否需要更新？
- [ ] 债务日志是否需要更新？
- [ ] 交接文档（`docs/01-guides/HANDOVER.md`）是否需要同步更新？
- [ ] ADR（`docs/05-decisions/README.md`）是否需要新增或更新？

### 1.5 安全层面

- [ ] 是否有敏感信息可能泄露（API Key、密码）？
- [ ] 是否有未鉴权的操作？
- [ ] 日志中是否会打印密钥等敏感信息？

### 1.6 极端场景层面（实盘红线）

> 为什么需要这个：审计发现 P0 问题 31 项、P1 问题 27 项，
> 但债务系统为 0 条——审视清单缺乏实盘场景视角是根本原因。
> 此层面用于捕获"代码现阶段能跑，但在边缘条件下必然出问题"的风险。

- [ ] **"最坏情况"推演**：所有数据源同时挂掉时，系统行为是否还有意义？
- [ ] **"用户误操作"推演**：输入非法代码、高频刷新、重复提交，系统是否会崩溃？
- [ ] **"悄悄变坏"推演**：数据格式正确但内容为空/过期，用户能发现吗？
- [ ] **"外部依赖全断"推演**：LLM API 不可用 + 数据源全挂 + 缓存清空，最小可用状态是什么？
- [ ] **"静默数据丢失"推演**：磁盘满、写入失败、缓存过期，是否会导致用户看到过期数据？
- [ ] **以上推演中发现的任何盲区，必须登记为债务** ← 强制转换链

## 2. 突发情况处理

### 2.1 遇到阻塞

```markdown
1. 记录阻塞原因到 AI 工作日志
2. 如果是代码问题 → 评估是否要新增债务
3. 如果是外部依赖（API 不可用等）→ 记录等待
4. 明确告知用户阻塞了什么，建议下一步
```

### 2.2 上下文耗尽

> 详细自动交接流程见 `CLAUDE.md`「上下文耗尽自动交接」节。

```markdown
1. **立即停止**新工作推进，执行会话结束 checklist（CLOSING.md §1）
2. Batch Loop 模式下：
   - 记录当前 Batch 做到哪一步（原子 todo 完成进度）
   - 卡住的决策点是什么
   - 下一步应该从哪开始
3. Git 提交，commit 信息含 `context-exhausted-handover` 标记
4. 在日志末尾标记「上下文耗尽，需续接」
5. 输出提示：「上下文已耗尽，交接文档就绪，开新会话后执行 /resume-session 继续」
```

### 2.3 发现之前的决策有误

```markdown
1. 更新相关 ADR（标记为 ⏪ 已替换）
2. 创建新的 ADR 记录新的决策
3. 在 AI 工作日志中说明变更理由
4. 更新受影响的代码和文档
```
