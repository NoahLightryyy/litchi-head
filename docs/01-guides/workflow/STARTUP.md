## 1. 核心原则

```
┌─────────────────────────────────────────────┐
│  每次 AI 会话 = 一次完整的工作单元           │
│                                             │
│  1️⃣ 启动时：理解上下文 + 加载最新状态        │
│  2️⃣ 工作中：代码 + 文档 + 测试同步推进       │
│  3️⃣ 结束时：更新日志 + 更新债务 + 提交记录    │
└─────────────────────────────────────────────┘
```

### 1.1 五同步原则

任何变更必须同步更新五样东西：

| # | 要素 | 说明 |
|---|------|------|
| 🖥️ | **代码** | 功能实现 |
| 🧪 | **测试** | 新功能必须有测试覆盖（目标 80%+） |
| 📝 | **文档** | README / 设计文档 / ADR / 部门文件 |
| 🐛 | **债务日志** | 新增债务或更新状态 |
| 🧹 | **引用清理** | 创建新方案后搜索旧方案在全项目的所有引用并逐处清理，不留下孤立的旧文件/旧命令/旧路径 |

> 只改代码不改文档 = 留下新债务。
> 只改文档不改代码 = 空谈。
> 改了代码不补测试 = 债务正在累积。
> **写了新工具不清理旧引用 = 流程有洞等你补。**

### 1.2 优先序原则

处理任务时按此优先级：

```
安全/数据完整性 → 阻塞项(S0) → 核心功能 → 可维护性 → 整洁优化
      🥇             🥈           🥉          🎯         💡
```

## 2. 部门工作模式总览

> **核心理念**：同一个 AI，进入不同代码目录 = 戴上不同部门的帽子。
> 在 `src/data/` 我是数据管道架构师，在 `frontend/` 我是前端专家——角色由部门文件夹定义。

### 工作模式图

```
                              ┌──────────────────────────────────────────────────┐
                              │              👤 你（产品负责人）                    │
                              │  给方向："修数据模块" / "加个辩论功能" / "调样式"    │
                              └────────────────┬─────────────────────────────────┘
                                               │ 一句话方向
                                               ▼
┌──────────────────────────────────────────────────────────────────────────────────┐
│                     🤖 AI（执行层 — 10 部门系统）                                 │
│                                                                                  │
│  ① 接收方向 → 判断属于哪个部门                                                     │
│     ↓                                                                           │
│  ② 加载 docs/06-departments/{id}/ROLE.md                                         │
│     → 进入角色（数据管道架构师 / 辩论系统专家 / 前端工程师 ...）                       │
│     ↓                                                                           │
│  ③ 加载 STANDARDS.md → 记住本部门技术红线                                          │
│     ↓                                                                           │
│  ④ 加载 HANDOVER.md → 了解当前状态和债务                                           │
│     ↓                                                                           │
│  ┌──────────────────────────────────────────────────────────────────────┐       │
│  │  ④' 跨部门？ → 同时加载涉及部门的 STANDARDS.md                          │       │
│  │  如"辩论结果 API 慢" → debate-engine + backend-api 两部会审             │       │
│  └──────────────────────────────────────────────────────────────────────┘       │
│     ↓                                                                           │
│  ⑤ 按 STANDARDS.md 规范开始工作                                                  │
│     ↓                                                                           │
│  ⑥ 遇到决策点 → 停下来问你                                                       │
│     ↓                                                                           │
│  ⑦ 完成 → 问你审不审                                                             │
│     ↓                                                                           │
│  ⑧ 会话结束 → 更新部门 HANDOVER.md + DEBT.md                                     │
│     ↓                                                                           │
│  ⑨ CI 健康自检 → ruff + pyright + pytest [标准](../ci/CHECKS.md)                                     │
│     ↓                                                                           │
│  ⑩ 写工作日志 + 提交                                                            │
└──────────────────────────────────────────────────────────────────────────────────┘
```

### 角色加载规则

| 场景 | 加载哪个角色 | 示例 |
|:-----|:------------|:-----|
| 只改 `src/data/` 的代码 | `01-data/ROLE.md` | 增加新数据源 |
| 只改 `src/debate/` 的代码 | `02-debate-engine/ROLE.md` | 修改编排节点 |
| 只改 `frontend/` 的代码 | `09-frontend/ROLE.md` | 修改组件 |
| 改 `backend/` 的路由 | `08-backend-api/ROLE.md` | 新 API endpoint |
| 跨 `src/data/` + `backend/` | 主角色 + 另一个的 STANDARDS | 改数据模型 + API 响应 |
| 跨 `src/debate/` + `src/data/` | 辩论部 + 数据部的 STANDARDS | 辩论发现数据不够用 |
| 架构级变更（全代码库） | 全部 10 个 STANDARDS | 改 LLM 调用方式 |
| 你只说"修一下问题" | 先问清楚哪个模块 → 再定角色 | — |

### 每个部门的 4 件套

```
docs/06-departments/{id}/
├── ROLE.md       ← 👤 角色定义：我是谁、边界、对外接口、变更通知谁
├── STANDARDS.md  ← 📐 技术规范：代码规范、测试要求、性能基线、审查清单
├── HANDOVER.md   ← 📋 工作交接：当前状态、开放债务、下一步优先级
└── DEBT.md       ← 🐛 债务清单：本部门专属债务
```

> **五同步原则升级**：代码 + 测试 + 文档 + 部门 DEBT.md/HANDOVER.md + **引用清理** 同步更新。

### 重要原则

| 原则 | 说明 |
|:-----|:------|
| **不进不出** | 不进部门文件夹，我就是一个中立的 AI。你说"去修数据模块"，我才加载角色 |
| **边界清晰** | 每个 ROLE.md 的"我不管什么"节明确划出界限——数据部不碰辩论逻辑 |
| **连坐变单坐** | 债务按部门管理——基础设施部的债务不会出现在数据部的文件里 |
| **变更通知制** | 改模型字段 → 看 ROLE.md 的"对外接口"节 → 通知所有消费者部门 |

## 3. 重大决策调研原则

遇到涉及架构选型、模块新增、技术方案对比的重大决策时，遵循以下流程：

### 重大决策三步法

1. 🔍 **先调研** — 搜索市场中同类竞品/成熟框架的做法（至少 2-3 个来源）
2. 📊 **对照分析** — 提炼各方案的核心设计 + 适用场景 + 优缺点
3. 🧑‍💻 **由人决策** — 将调研结论（含推荐方案）呈现给开发者，由人做最终选择

### 什么时候必须调研

- 新增一个完整模块（数据库、记忆系统、缓存层等）
- 涉及存储/基础设施选型（存什么、怎么存）
- 系统间的集成模式（同步/异步/事件驱动）
- 任何改动后难以回退的决策

### 原则

- 调研不意味着照搬——根据本项目的实际情况（技术栈、阶段、团队）做适配
- 竞品做的好的地方要借鉴，做的不好的要避开
- 调研结果必须记录到 ADR 的「舍弃方案」或「备选方案」部分
- 最终决策权在开发者，AI 提供分析但不代为决断

## 4. 会话启动标准流程

每次 AI 开始新会话时，按此顺序加载状态：

### 4.1 第 1 步：检查项目状态

> **上下文优化**：核心信息（身份卡/ADR/当前状态）已由 Memory 系统自动注入，
> 无需再逐份完整读取 CLAUDE.md、债务日志和设计文档。

```markdown
### 自动注入（不占手动读取）
- [ 自动] project-identity → 技术栈、红线、定位（memory/）
- [ 自动] architecture-decisions → ADR 要点（memory/）
- [ 自动] current-state → 活跃任务、模块完成度（memory/）

### 快速同步（启动时读取，~3-5K token）
- [ ] 阅读 `docs/01-guides/HANDOVER.md` §部门一览 + §当前跨部门优先级
- [ ] 阅读 `docs/04-changelog/logs/README.md` 了解最近工作索引
- [ ] 阅读最新工作日志（前次会话做了什么）

### 快速检查
- [ ] 检查当前 git 状态（是否有未提交变更）
- [ ] 确认 `.env` 配置是否完整（API Key 等）
- [ ] 确认 `.gitignore` 包含 `.env`（防误上传）
- [ ] 确认 `.env` 未被 Git 追踪（`git check-ignore .env`）
- [ ] 根据任务类型启用对应子代理（详见 [DEVELOPMENT.md](DEVELOPMENT.md) §Agent 编排）

### 按需加载（用到时才读）
- [按需] `docs/01-guides/debt/ROUTER.md` — 要修债务时再读
- [按需] `docs/00-overview/ROADMAP.md` 项目看板 — 要查全局进度时
- [按需] `docs/99-archive/` 设计文档 — 要修改架构时
- [按需] [DEVELOPMENT.md](DEVELOPMENT.md) — 日常开发流程
- [按需] [CLOSING.md](CLOSING.md) — 会话结束流程
- [按需] [EMERGENCIES.md](EMERGENCIES.md) — 审视/突发情况
- [按需] [CI 治理体系](../ci/README.md) — 要查 CI 状态、修 CI 时
```

### 4.2 第 2 步：理解上下文

```markdown
- [ ] 明确本次会话的目标任务
- [ ] 确认依赖的模块是否已经就绪
- [ ] 如果涉及跨模块变更，先确认接口契约（Pydantic 模型）
- [ ] 如果涉及技术选型，先查阅 ADR 避免重复决策
```

## 5. 部门角色加载

> **这是部门体系的核心步骤**。进入工作前先确定"我在哪个部门工作"，加载对应的角色身份。

```markdown
### 部门角色加载

#### ① 确定主部门
本次任务主要修改哪个代码目录？

| 代码目录 | 对应部门 |
|:---------|:---------|
| `src/data/` | 🗄️ 01-数据管道部 |
| `src/debate/` + `src/risk/` | 🎯 02-辩论引擎部 |
| `src/agents/` + `src/core/` | 🤖 03-AI Agent 架构部 |
| `src/memory/` | 🧠 04-记忆系统部 |
| `src/risk/`（如独立修改）| 🛡️ 05-风控管理部 |
| `src/trader/` | 💹 06-交易执行部 |
| `src/backtest/` | 🔬 07-回测研究部 |
| `backend/` | 🌐 08-后端 API 部 |
| `frontend/` | 🎨 09-前端部 |
| `src/utils/` | ⚙️ 10-基础设施部 |
| 跨多个目录 | 🔄 00-跨部门 |

#### ② 加载 ROLE.md
- [ ] 阅读 `docs/06-departments/{id}/ROLE.md`
  - [ ] 记住"我管什么 / 不该管什么"的界限
  - [ ] 查看"对外接口"节——我提供什么、依赖谁、变更通知谁
  - [ ] 记住质量标准

#### ③ 加载 STANDARDS.md
- [ ] 阅读 `docs/06-departments/{id}/STANDARDS.md`
  - [ ] 记住代码规范示例（✅ 做什么 / ❌ 禁止什么）
  - [ ] 记住测试要求（必须覆盖的场景、Mock 策略）
  - [ ] 记住文件大小红线、性能基线

#### ④ 加载 HANDOVER.md
- [ ] 阅读 `docs/06-departments/{id}/HANDOVER.md`
  - [ ] 了解当前状态和开放债务
  - [ ] 确认"下一步优先级"——要不要顺便修个债务

#### ⑤ 跨部门检查
- [ ] 是否涉及其他部门的代码？→ 加载其 STANDARDS.md
- [ ] 是否改了数据契约（Pydantic 模型）？→ 看 ROLE.md 的"变更通知"节
- [ ] 是否需要其他部门配合？→ 记录需要通知的部门
```

> **一键启动**：简单任务（只改一个部门代码）→ 5 步约 10 秒。
> 复杂任务（跨部门）→ 多花 30 秒加载涉及部门的 STANDARDS。
> **纪律**：不加部门角色直接开工 = 裸奔。不要裸奔。
